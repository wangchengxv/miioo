# 项目 CLAUDE 规则

## 1. 仓库定位

本仓库不是从零搭建的新系统，而是在既有前端页面、后端接口和 Harness 文档体系基础上，持续做前后端打通、生产化收口和工程规范固化。

当前统一口径：

- 生产后端代码：`backend/`
- 生产前端代码：`frontend/`
- 文档事实与治理层：`docs/`

## 2. 事实源优先级

按以下顺序判断事实：

1. 当前任务直接涉及的代码与配置
2. `docs/architecture/`、`docs/decisions/`、`docs/plans/`
3. `backend/BACKEND_API_DOC.md`
4. `frontend/PROJECT.md`
5. 历史归档文档

若代码与文档冲突，必须在结果中指出冲突，并把修正意见回写到文档。

## 3. 代码工作原则

- 保留前端现有 UI 设计和交互设计，不把联调任务扩散成视觉重构任务。
- 前端页面层只消费视图模型，不直接承载后端字段翻译。
- 前端请求、字段兼容、媒体 URL 拼接、任务轮询统一收口在 `frontend/src/api/` 与工具层。
- 后端优先修契约、配置、任务状态、媒体链路和部署边界，不随意改动已稳定业务语义。
- 新增规则优先固化到文档、脚本或配置中，避免重复犯错。

## 4. 文档工作原则

- 所有阶段性结论优先写入 `docs/plans/` 小文档，不把入口文档无限膨胀。
- 架构边界写入 `docs/architecture/`，决策写入 `docs/decisions/`，执行计划写入 `docs/plans/`，运行手册写入 `docs/runbooks/`。
- 每次任务完成后，至少检查是否需要更新：
  - `docs/plans/module-progress.md`
  - `docs/plans/项目进度文档.md`
  - `frontend/PROJECT.md`
  - `CHANGELOG.md`

## 5. 执行禁区

- 不擅自改动前端整体视觉风格、页面布局骨架和核心交互路径。
- 不把页面临时兼容逻辑散落到多个组件里。
- 不删除历史 SQL、素材、上传文件、参考代码和归档文档。
- 不在未确认的情况下把 `docs/` 里的历史口径直接当成已自动生效的运行代码。

## 6. 当前默认任务主线

当前默认主线不是“新做页面”，而是：

1. 以原系统事实为基线梳理生产需求
2. 以 `backend/` 为生产后端承接真实能力
3. 以新 `frontend/` 为生产前端承接页面与适配层
4. 以 Harness 文档体系承接决策、进度、回写与交接
