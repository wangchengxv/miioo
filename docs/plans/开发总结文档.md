## 1. 文档定位

本文档用于沉淀当前项目已经被多轮实践验证过的开发经验、治理方法、排障结论和长期有效的工程判断。

它不是逐日改动流水账，也不是专项计划集合，而是一份供后来者快速少踩坑的“经验总稿”。

## 2. 总体开发结论

当前项目最重要的开发结论只有一句话：

不要为了让接口“看起来更顺手”去破坏前端页面语义，应保持前端产品事实源不动，把联调复杂度集中收口到 `src/api/`、后端任务制和文档治理层。

围绕这句话，当前仓库已经形成了比较稳定的工程方法：

- 页面层不直接适配后端字段
- 适配层负责双端翻译
- 后端负责真实持久化、任务和错误语义
- 文档负责留痕和交接

## 3. 已被验证有效的工程方法

### 3.1 前端优先，适配层承接

这是当前项目最有效的总体方法。

原因：

- 页面产品语义稳定，不必频繁跟随接口字段改名
- 多页面共用的命名冲突能在适配层集中治理
- 后续接口字段变化只影响一层，而不是散落到组件树

适用场景：

- `desc -> description`
- `ratio -> aspect_ratio`
- `style -> visual_style`
- `shot -> storyboard`
- `char -> character`

### 3.2 小步闭环优于一次性大重构

实践证明，当前仓库最有效的推进方式是：

- 先打通最短真实链路
- 再补状态承接
- 再补异常路径
- 最后回写文档

如果一开始就想同时重构 UI、接口、状态和模型能力，反而更容易失控。

### 3.3 文档治理必须和代码同步

当前仓库已经反复证明：

- 只改代码不回写文档，后续就会重复踩坑
- 只写大母本文档，不拆当前层小文档，后续查找成本会持续上升

最有效的方式是：

- 当前结论优先写入小文档
- 历史母本只在必要时补录

## 4. 前端联调经验

### 4.1 页面层不要直接写真实请求

一旦页面直接请求后端，后果通常包括：

- 后端字段散落在页面和组件内
- 后续接口变化需要大面积回归
- 同一概念在多个页面重复翻译

因此当前项目必须坚持：

- 页面只调用 `src/api/`
- 页面不感知后端蛇形命名、兼容字段、上传前置细节

### 4.2 最小状态承接比大范围改 UI 更重要

在联调过程中，真正影响是否“看起来通了”的，常常不是 UI 漂不漂亮，而是页面是否承接了真实业务状态：

- 加载中
- 失败态
- 空态
- 任务中间态
- 刷新后的回显状态

因此当前前端改动应优先补真实状态承接，而不是重做样式。

### 4.3 模型能力必须统一收口

创作页、主体页、分镜页都反复出现过同类问题：

- 不同模型支持的参考图数量不同
- 时长 / 分辨率范围不同
- 支持的生成方式不同

经验结论是：

- 模型能力不能由页面自己猜
- 必须统一收口到能力表和适配层

## 5. 后端联调与任务制经验

### 5.1 长任务必须任务化

实践已经证明，对于主体提取、智能分镜、图片生成、视频生成、导出等长链路，任务制比同步等待整包返回更可靠。

统一经验：

- 长任务先返回 `taskId`
- 前端轮询任务状态接口
- 页面切换后任务继续执行
- 终态必须明确支持 `partial`

### 5.2 错误信息必须给到可诊断 detail

后端如果只返回笼统 `500` 或 `502`，前端无法正确展示，排障也很低效。

当前工程经验是：

- router 层应尽量给出明确 `detail`
- 对上游模型错误、参数错误、媒体不可达错误要保留可诊断语义

### 5.3 历史调试埋点要及时清理

当前仓库已经踩过一次性调试代码长期残留的坑，例如：

- 临时端口
- 临时 event 推送
- 一次性 print / debug 逻辑

经验结论是：

- 排障临时注入一旦验证结束应立即清理
- 否则会给后续联调制造新的噪音

## 6. 媒体链路经验

### 6.1 blob 只做临时预览

实践证明，本地 blob URL 只能用于页面预览，不能作为业务事实源。

统一规则：

- 本地文件先上传
- 后端保存为受管资源
- 业务接口消费真实 URL

### 6.2 视频 / 音频参考最容易出问题

相比图片，视频和音频参考更容易因为以下原因失败：

- 本地地址
- 私网地址
- HTTP 地址
- 上游不可访问

经验结论：

- 页面不直接把本地文件交给外部模型
- 后端必须做公网可访问 URL 转换和必要重托管

### 6.3 原图、缩略图、下载图必须分语义

资产库和创作页反复证明：

- 展示图
- 卡片缩略图
- 原始图
- 下载图

如果不区分，会导致回显、下载、详情弹窗、性能优化全部混乱。

### 6.4 展示错了，通常先错在字段语义

最近几轮分镜、资产库和剪辑素材池问题进一步证明：

- 黑图、破图、空白卡片，很多时候不是 CSS 问题
- 真正的根因往往是页面吃错了字段，或把错误类型的 URL 塞进了错误组件

典型错误包括：

- 把视频地址当图片地址传给 `<img>`
- 把缩略图字段和原文件字段混用
- 列表、详情、下载分别读取不同来源，导致“看得到但点不开”

经验结论：

- 遇到展示问题时，先查字段语义，再查样式
- 页面是否“有值”不代表结果“拿对了”
- 一旦字段语义混乱，局部补样式只会掩盖问题

### 6.5 列表、详情、下载必须共享同一视图模型

资产库、创作页和分镜页已经多次证明：

- 如果卡片使用一套字段
- 详情弹窗使用另一套字段
- 下载按钮再自己猜第三套字段

最后一定会出现展示不一致。

更稳定的做法是：

- 由 `src/api/` 统一构造视图模型
- 列表、详情、预览、下载都从同一模型读取
- 回退顺序只定义一次，不在多个组件里重复写

### 6.6 长任务展示必须先承认“结果不是同步拿到的”

图片生成、视频生成、导出等链路已经反复证明：

- 如果页面默认假设“提交成功 = 最终结果已拿到”
- 那就会频繁出现空 URL、错误提示和假失败

经验结论：

- 长任务展示要先承认结果是异步回流的
- 提交接口只负责返回任务身份
- 页面真正展示结果，要等任务终态后刷新真实详情或列表

### 6.7 展示残影往往是缓存问题，而不是渲染问题

账号切换、项目切换、生成后列表不刷新等问题已经说明：

- 很多“显示错了”并不是渲染逻辑错了
- 而是页面还在展示上一用户、上一项目或写前缓存的旧结果

经验结论：

- 展示链路必须把缓存隔离和写后失效纳入设计
- 用户作用域、项目作用域和任务终态刷新，都是展示正确性的组成部分

## 7. 文档治理经验

### 7.1 大母本保留，但不要继续膨胀

当前项目已经有多个历史大文件：

- `PROJECT.md`
- 全局 `进度文档.md`
- `HARNESS_P0_HISTORY.md`

经验结论是：

- 大母本保留历史价值
- 但新增事实应优先进入当前层小文档

### 7.2 入口一定要明确

只要文档入口不清楚，后来者就会再次回到“全局搜索 + 猜哪份最新”的状态。

因此当前最有效的做法是：

- 根级由 `HARNESS_DOC_INDEX.md` 总导航
- `docs/README.md` 做文档中心入口
- 当前层文档承担真正日常阅读入口

### 7.3 职责必须去重

如果同一类内容同时存在于：

- 需求文档
- 路线图
- 进度文档
- backlog
- changelog

那就迟早会不同步。

经验结论：

- 每份文档只负责一种主要职责
- 新文档应以汇总和跳转为主，不重复抄写完整接口明细

## 8. 典型排障经验

### 8.1 长任务超时与中断感

当前已有经验表明：

- 如果同步等待 LLM 或视频生成整包返回，极易超时
- 即使任务实际还在执行，页面也会给用户“已经失败”的错觉

解决方式：

- 改为任务制
- 页面轮询状态
- 中间结果及时回显

### 8.2 数据库迁移状态漂移

资产页和创作页的 `500` 问题已经证明：

- 本地数据库迁移状态和代码事实漂移，会造成看似“前端页面坏了”的问题
- 实际上根因可能是 migration 状态损坏或字段缺失

因此排查时不要只看页面和网络层，也要尽快核对：

- Alembic 版本
- 表字段是否齐全
- 本地库是否已经 upgrade 到正确版本

### 8.3 供应商路由误判

模型名、provider、能力表如果没有统一事实源，就容易出现：

- 页面以为选的是 A
- 后端实际走的是 B
- 文档写的又是 C

结论：

- 模型能力和 provider 归属必须统一落库并由接口返回
- 页面不能靠模型名字符串猜供应商

## 9. 目前仍需长期观察的问题

以下问题虽已有方向，但仍需持续观察：

- 真实 provider 环境下的稳定性
- 任务 `partial` 和失败提示的一致性
- 视频 / 音频参考链路的公网访问和 HTTPS 收口
- 聚合页是否需要更多后端聚合接口
- 分镜专项的新字段口径是否已被所有下游真正统一消费
