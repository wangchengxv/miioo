# 微信二维码登录前端实现方法

> 类型：plans / 实现方法沉淀
> 日期：2026-06-23
> 来源：`前端改动记录.md` 中 `2026-06-23` 的微信二维码登录改动条目
> 关联代码：`miioo-feat-project/frontend/src/api/auth.js`、`miioo-feat-project/frontend/src/components/LoginModal.jsx`、`miioo-feat-project/frontend/src/components/WechatOfficialQr.jsx`、`miioo-feat-project/frontend/src/utils/serialPolling.js`、`miioo-feat-project/frontend/src/pages/Home.jsx`

## 1. 提取出的改动字段

### 1.1 基础字段

- 项目：`miioo-feat-project/frontend`
- 主题：补齐真实微信二维码登录前端闭环，适配当前后端微信开放平台扫码登录接口

### 1.2 改动文件

- `miioo-feat-project/frontend/src/api/auth.js`
- `miioo-feat-project/frontend/src/components/LoginModal.jsx`
- `miioo-feat-project/frontend/src/components/WechatOfficialQr.jsx`
- `miioo-feat-project/frontend/src/utils/serialPolling.js`
- `miioo-feat-project/frontend/src/pages/Home.jsx`

### 1.3 改动思路

- 先把前端 API 层从旧版字段假设收口到当前后端真实契约，避免组件层继续猜 `session_id / qr_code_value / bind_token`。
- 再把登录弹窗改成消费真实微信授权链接，不再把二维码当作静态图片资源，而是接入微信官方 `WxLogin` 组件。
- 再补扫码后的状态承接，包括轮询、`postMessage` 同步、需要绑手机号时的继续处理。
- 最后在首页接住 `?code=&state=` 回调，把根路径回调闭环补完整，避免“扫码后只回首页、不登录”。
- 展示层仅做轻量微调，二维码视觉位置整体上移 `8px`，不影响扫码逻辑。

## 2. 实现目标

这次实现要解决的不是“多一个微信按钮”，而是把微信扫码登录做成一个完整闭环：

1. 前端向后端申请微信二维码会话。
2. 后端返回真实微信授权链接。
3. 前端用微信官方组件渲染二维码。
4. 用户扫码确认后，微信回跳到 `https://miiooai.com/?code=...&state=...`。
5. 首页拿 `code + state` 调后端完成回调。
6. 登录弹窗通过轮询或 `postMessage` 收到结果。
7. 已绑定手机号则直接登录；未绑定手机号则继续手机号绑定流程。

## 3. 具体实现方法

## 3.1 API 适配层收口

文件：`src/api/auth.js`

实现方法：

- 新增 `apiCompleteWechatCallback({ code, state })`，对应后端回调完成接口。
- 新增内部 `apiConfirmWechatLogin(...)`，统一承接“扫码确认后绑定手机号”场景。
- 把 `apiGetWechatQrCode()` 从旧的静态二维码 URL 思路改成消费后端返回的真实微信授权链接，并映射成前端当前使用的字段：
  - `session_id -> qrcode_id`
  - `qr_code_value -> raw_qr_code_value`
  - `expires_in -> expire_seconds`
- 把 `apiPollWechatQrCodeStatus()` 收口到当前后端真实轮询路径。
- 把 `apiBindMobileWithBindToken()` 收口为“继续调用后端确认登录接口”，而不是再假设一个旧版独立绑定接口。

这样做的原因：

- 页面层和弹窗层不应该直接理解后端所有字段语义。
- 先在 API 适配层完成翻译，组件只消费稳定视图模型，后续接口再变也只改一层。

## 3.2 登录弹窗改为官方二维码模式

文件：`src/components/LoginModal.jsx`

实现方法：

- 引入 `WechatOfficialQr` 组件，替代“直接把 `qrcode_url` 当图片展示”的旧方案。
- `loadQrCode()` 负责拉取二维码会话：
  - 调 `apiGetWechatQrCode()`
  - 保存 `qrcode_id`
  - 保存 `raw_qr_code_value`
  - 进入 `ready` 状态并启动轮询
- `startPolling()` 负责持续查询扫码状态：
  - `confirmed`：直接登录成功
  - `need_bind_mobile`：进入手机号绑定流程
  - `expired`：二维码过期
  - `scanned`：显示“已扫码，待确认”
  - `error`：显示错误提示
- 补入 `handleWechatCallbackMessage`，让首页回调完成后可以通过 `postMessage` 主动通知弹窗刷新状态，而不必完全等轮询。

这样做的原因：

- 微信二维码登录本质是一个异步状态机，不是一次性接口。
- 弹窗必须同时支持“轮询兜底”和“回调主动通知”，否则体验不稳定。

## 3.3 封装官方微信二维码组件

文件：`src/components/WechatOfficialQr.jsx`

实现方法：

- 动态加载微信官方脚本：`https://res.wx.qq.com/connect/zh_CN/htmledition/js/wxLogin.js`
- 从后端返回的授权链接中解析：
  - `appid`
  - `redirect_uri`
  - `state`
  - `scope`
- 通过 `new WxLogin(...)` 真正渲染二维码。
- 使用 `self_redirect: true`，让微信扫码后的回跳发生在当前二维码 iframe 内。
- 用 `scale` 控制二维码在裁切框中的缩放，用 `offsetY=-8` 做轻量视觉上移。

这样做的原因：

- 官方二维码不是普通图片，而是微信脚本生成的登录部件。
- 直接渲染授权链接对应的二维码，才能保证扫码后状态和 `state` 与后端会话一一对应。

## 3.4 抽公共轮询工具

文件：`src/utils/serialPolling.js`

实现方法：

- 封装 `createSerialPolling(...)`，保证轮询任务串行执行。
- 支持页面隐藏暂停：`pauseWhenHidden = true`
- 支持错误退避：`errorBackoffMultiplier`
- 支持连续错误上限：`maxConsecutiveErrors`
- 支持外部停止：`stop()`

这样做的原因：

- 登录弹窗里如果直接 `setTimeout + fetch` 很容易重复请求或并发轮询。
- 抽成公共串行轮询后，二维码状态轮询更稳，也更容易复用。

## 3.5 首页承接根路径回调

文件：`src/pages/Home.jsx`

实现方法：

- 新增 `readWechatCallbackParams()`，从 `window.location.search` 读取 `code / state`。
- 新增 `clearWechatCallbackParamsFromUrl()`，在处理完成后清理 URL 中的微信参数，避免刷新后重复执行。
- 新增 `postWechatCallbackMessage(payload)`，在 iframe 场景下把处理结果回传给父窗口。
- 页面初始化时检测 `?code=&state=`：
  - 参数缺失：提示错误并清理 URL
  - 参数完整：调用 `apiCompleteWechatCallback({ code, state })`
  - 成功后根据状态决定是发 `postMessage`，还是在非 iframe 情况下直接给出提示

这样做的原因：

- 当前后端把微信回调地址归一到根路径 `/`，所以首页本身必须能吃回调参数。
- 如果首页不承接回调，扫码只会“返回首页”，不会真正完成登录链路。

## 4. 文件职责拆分

| 文件 | 职责 |
|---|---|
| `src/api/auth.js` | 契约适配与字段翻译 |
| `src/components/LoginModal.jsx` | 扫码入口、状态展示、绑定手机号流程 |
| `src/components/WechatOfficialQr.jsx` | 官方二维码脚本加载与渲染 |
| `src/utils/serialPolling.js` | 串行轮询与错误退避 |
| `src/pages/Home.jsx` | 根路径回调参数承接与结果通知 |

## 5. 实现顺序建议

如果后续要在其它前端工程里复刻这套能力，建议按下面顺序落地：

1. 先对齐后端契约，明确二维码创建、轮询、回调完成、手机号绑定这四类接口。
2. 再实现 `src/api/auth.js` 的适配层，先把字段翻译稳定。
3. 再接入官方 `WxLogin` 组件，确认二维码能正常展示。
4. 再补 `LoginModal.jsx` 的扫码轮询与绑定分支。
5. 最后补首页对 `?code=&state=` 的承接与 URL 清理。
6. 完成后再微调视觉位置，不要先改样式再补逻辑。

## 6. 验收口径

- 二维码可以正常拉起，且不是静态占位图。
- 手机扫码后，状态能从 `pending` 正确推进到 `scanned / confirmed / need_bind_mobile`。
- 已绑定手机号用户可以直接完成登录。
- 未绑定手机号用户可以继续绑定手机号并完成登录。
- 首页回调参数处理后会被清理，不会因刷新重复执行。
- 二维码位置可通过 `offsetY` 做轻量视觉调整，不影响扫码能力。

## 7. 方法总结

这套实现的核心不是“多写几个接口函数”，而是坚持三个收口原则：

- 契约收口在 `src/api/auth.js`
- 状态收口在 `LoginModal.jsx + serialPolling.js`
- 回调收口在 `Home.jsx`

只要这三个收口点不再散落，后续无论是回调路径调整、字段命名变化，还是视觉微调，都能在最小范围内完成修改。
