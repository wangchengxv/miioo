# 生产环境并发与性能治理计划

## 1. 文档定位

本文档用于把当前仓库已经识别出的“生产环境高并发、大量 API 请求、长任务、媒体流量”风险，统一收口为一份可执行的 Harness 工程治理计划。

它负责：

- 固化当前代码事实下已经暴露的并发与性能风险
- 把风险点转成明确的工程约束、分阶段任务和验收标准
- 为后续后端、前端、部署、文档回写提供统一事实源

它不负责：

- 直接替代详细实现设计
- 代替真实环境压测报告
- 代替上线前人工联调结论

## 2. 当前代码事实

基于当前仓库代码，生产运行时已经具备以下事实：

- 后端生产脚本默认使用多 worker `uvicorn` 启动，当前默认 `WORKERS=2`
- 数据库连接池默认值为 `pool_size=5`、`max_overflow=10`
- 验证码、扫码登录会话、手机号换绑验证码仍存在进程内内存态
- 多类长任务仍由 Web 进程内的 `asyncio.create_task(...)` 直接调度
- `FastAPI` 当前直接挂载 `/uploads` 提供静态媒体
- 仓库已存在 SSE / `StreamingResponse` 场景
- 前端请求层已有读缓存、并发去重和 token 刷新能力
- 但前端仍存在页面/组件直接 `fetch`、高频轮询、资源直拉下载等绕过统一适配层的场景
- 当前仓库已开始补热点列表分页、慢请求日志和热点索引，但生产运行时治理尚未形成完整闭环

## 3. 当前主要风险

### 3.1 共享状态风险

- 多 worker 部署下，进程内 `dict` 状态不共享
- 验证码、扫码登录状态、手机号换绑状态可能因请求落到不同 worker 而丢失
- 这类问题不只是慢，而是会直接表现为偶发失败、状态不一致、线上难复现

### 3.2 Web 进程挤占风险

- 长任务仍与普通 API 共用 Web 进程事件循环
- 任务数量增加时，普通读写接口、SSE、轮询接口会被一起拖慢
- 进程重启或 worker 被回收时，任务存在中断或丢失风险

### 3.3 轮询风暴风险

- 前端当前存在 `2s`、`3s` 级别轮询
- 多页面、多弹窗、多用户叠加后，会把实际业务量放大成高 QPS
- 页面层直连请求还会绕过统一缓存、去重、限流和失效策略

### 3.4 数据库与连接预算风险

- 连接池参数是“每个 worker 一套”，worker 数量增加会同步放大数据库连接占用
- 高热点列表若缺少匹配索引，会在数据量上来后出现排序、分页和过滤变慢
- 若缺少统一容量预算，容易出现“应用层加 worker，数据库先耗尽连接”的反效果

### 3.5 媒体与带宽风险

- `/uploads` 当前由应用层直接承接
- 图片、视频、音频预览和下载一旦并发升高，会直接占用应用层带宽、文件 IO 和连接数
- 前端资源下载若仍在页面层直接 `fetch(url)`，将继续放大重复流量

### 3.6 SSE 与上游调用风险

- SSE 和流式响应会长期占用连接、worker 和反向代理资源
- 当前大量外部模型/媒体调用仍以局部创建 `httpx.AsyncClient` 为主
- 如果缺少统一超时、连接池、并发阈值和熔断策略，上游超时或限流会反向放大本系统抖动

### 3.7 可观测性不足风险

- 当前已经有慢请求和慢 SQL 基线，但仍缺完整的运行时指标闭环
- 若没有任务队列长度、轮询频率、SSE 在线数、媒体带宽、上游 429/5xx 统计，后续只能靠日志被动排障

## 4. Harness 强制约束

以下约束从本计划生效起，作为后续并发与性能治理的默认工程约束：

1. 多 worker 生产态禁止继续依赖进程内状态承接登录、验证码、轮询票据等共享运行时数据。
2. 新增长耗时任务时，禁止默认落到 Web 进程内 `asyncio.create_task(...)` 作为长期方案；必须先判断是否进入统一任务队列。
3. 页面层禁止新增直接 `fetch` / 资源直拉下载；所有网络访问继续统一收口到 `frontend/src/api/`。
4. 轮询类能力禁止散落在页面层各自实现；必须统一封装退避、停止条件、可见性暂停与共享状态策略。
5. 媒体展示必须区分 `thumbnailUrl / imageUrl / videoUrl / audioUrl / fileUrl / downloadUrl`，禁止继续让应用层承担本该由静态服务/CDN 处理的热点媒体流量。
6. 新增热点列表或聚合查询时，必须同步判断分页方式、排序稳定性、返回体大小与索引覆盖情况。
7. 新增 SSE / 流式接口时，必须同步说明连接上限、超时策略、回退机制与反向代理要求。
8. 任何已发现的性能问题，优先转成代码、配置、脚本、CI 或文档约束，不仅停留在聊天结论。

## 5. 目标分级

### 5.1 P0：生产稳定性基线

目标：

- 先消除会导致高并发下直接出错或随机失败的结构性问题

必须完成：

- 把验证码、扫码登录会话、手机号换绑验证码从进程内状态迁到 Redis 或等价共享状态层
- 梳理当前所有 `asyncio.create_task(...)` 入口，区分“可继续短任务化承接”和“必须外移到队列执行”的任务
- 为登录、验证码、扫码轮询、任务创建、任务轮询等接口补齐限流策略
- 把 `/uploads` 的生产承载方案从应用层剥离到 `Nginx` 静态托管，后续再视情况接 CDN/对象存储
- 收口前端页面层的直接 `fetch` 和分散轮询，统一挂到 `src/api/`

验收标准：

- 多 worker 下登录/绑定/验证码状态不再依赖单进程命中
- 热点轮询接口具备明确的频率上限和失败退避策略
- 大文件静态访问不再默认穿过 FastAPI 应用层

### 5.2 P1：容量与治理闭环

目标：

- 让系统从“能跑”进入“可预测扩容”

必须完成：

- 制定 Web worker、任务 worker、数据库连接、Redis、Nginx 的容量预算表
- 对高频列表、任务列表、通知列表、项目列表、资产列表、分镜列表完成索引与返回体审计
- 统一上游 HTTP 客户端策略：连接池、超时、重试、熔断、并发 semaphore
- 为轮询、SSE、任务执行、媒体下载补齐指标面板和日志维度
- 为高频热点接口补齐压测清单和性能基线记录

验收标准：

- 关键热点链路均有“请求量 -> 数据库 -> 上游 -> 返回体 -> 监控”闭环视图
- 可以明确回答“增加 worker 是否会打爆数据库连接”这类容量问题
- 至少有一版针对认证、任务、列表、媒体链路的压测结果留仓

### 5.3 P2：体验与成本优化

目标：

- 在稳定性达标后继续降低带宽、计算和等待成本

必须完成：

- 前端路由与重量模块做更明确的分包和懒加载治理
- 静态资源接入压缩、缓存头和必要的 CDN 策略
- 轮询场景中逐步评估是否以 SSE/WebSocket 替换高频短轮询
- 对图片、视频、音频详情与下载链路做缓存命中率优化

验收标准：

- 首屏、聚合页和高热点媒体页具备更稳定的加载时间
- 用户重复访问的静态资源命中率有明确提升
- 轮询请求量相较当前基线显著下降

## 6. 分项治理清单

### 6.1 共享状态治理

- 范围：登录验证码、扫码登录、手机号换绑、未来所有多 worker 共享票据
- 当前问题：进程内状态无法跨 worker 共享
- 约束：统一迁移到 Redis，并使用 TTL、幂等键、原子更新
- 交付物：共享状态实现、配置说明、失效策略文档、故障排查 runbook

### 6.2 任务运行时治理

- 范围：图片生成、视频生成、音频生成、分镜后台生成、导出任务等长任务
- 当前问题：Web 进程内直接起后台协程，运行时边界不清
- 约束：Web 进程只负责创建任务、查状态、取消；耗时执行迁到统一队列 worker
- 交付物：任务执行分层方案、重试策略、失败补偿、取消语义、任务容量预算

### 6.3 前端请求治理

- 范围：所有页面轮询、下载、详情刷新、通知读取、登录轮询
- 当前问题：仍有页面级直接请求与分散轮询
- 约束：统一通过 `src/api/` 管理缓存、去重、失效、轮询频率、退避和停止条件
- 交付物：请求封装收口清单、页面直连清理清单、轮询统一封装

### 6.4 数据库治理

- 范围：热点列表、聚合查询、任务表、资产表、分镜表、通知表、项目表
- 当前问题：连接预算未形成制度，部分高频条件需要继续复合索引化
- 约束：新增热点查询必须同时审计索引、分页、稳定排序、响应体大小
- 交付物：索引计划、连接预算表、压测记录、必要时引入 `PgBouncer` 方案说明

### 6.5 媒体与静态资源治理

- 范围：`/uploads`、图片/视频/音频预览、详情下载、首帧图、海报图
- 当前问题：应用层直接承担热点静态资源流量
- 约束：生产媒体优先由 `Nginx`/对象存储/CDN 承接，应用层只负责业务授权与元数据
- 交付物：静态资源托管方案、媒体访问路径说明、缓存策略、回源策略

### 6.6 SSE 与外部依赖治理

- 范围：LLM 流式对话、剧本生成流式输出、模型服务 HTTP 调用
- 当前问题：连接长占用、上游调用策略分散
- 约束：统一连接池、超时、并发阈值、错误分类、熔断和降级策略
- 交付物：SSE 运行约束、上游访问策略、代理层配置要求、超时矩阵

### 6.7 可观测性治理

- 范围：请求、SQL、队列、轮询、SSE、媒体、上游 API
- 当前问题：慢日志基线已有，但指标维度不足
- 约束：关键热点链路必须具备可量化指标，而非只看报错日志
- 交付物：指标清单、日志字段清单、告警阈值、排障流程

## 7. 实施顺序

### 阶段 A：基线排雷

- 共享状态 Redis 化
- 页面层直连请求盘点
- 热点轮询盘点
- `/uploads` 静态流量承载方案确认
- 任务入口分类：短任务/长任务/必须排队任务

### 阶段 B：运行时收口

- 队列化长任务
- 统一限流与并发控制
- 统一上游 HTTP client
- 数据库连接预算与索引补齐
- 关键热点接口压测首轮执行

### 阶段 C：观测与成本优化

- 指标面板与告警
- 前端分包、静态资源压缩与缓存
- 轮询退避优化
- SSE/轮询替代策略评估

## 8. 非目标

以下事项不属于本计划直接范围：

- 不以性能治理名义重做前端 UI 或交互
- 不在未确认收益的情况下盲目上复杂基础设施
- 不把真实环境联调替换成纯本地假压测结论
- 不因为追求高并发而破坏当前已经稳定的业务语义

## 10. 当前建议执行顺序

如果下一步开始真正落地，建议按以下顺序推进：

1. 先做共享状态 Redis 化与轮询收口，解决多 worker 下的随机失败问题
2. 再做长任务队列化与 `/uploads` 静态流量剥离，解决 Web 进程被挤占问题
3. 再做数据库容量预算、索引审计、上游调用治理，建立稳定扩容基线
4. 最后再做前端构建优化、缓存命中率优化和体验层优化
