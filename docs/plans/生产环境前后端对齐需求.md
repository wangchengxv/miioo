# 生产环境前后端对齐需求

## 1. 需求目标

本次任务的目标不是重做产品，而是在保留现有前端页面视觉和交互语义的前提下，完成生产环境前后端组合的重新对齐，形成一个可联调、可部署、可持续维护的完整工程。

当前统一口径如下：

- 生产后端：`/Users/xingyi/Documents/222222/backend`
- 生产前端：`/Users/xingyi/Documents/222222/frontend`
- 原系统事实基线：`/Users/xingyi/Documents/222222/docs`

## 2. 背景说明

- `docs/` 承接的是原系统前后端、架构、计划、路线图和 Harness 治理事实，不是新的业务代码目录。
- 当前 `frontend/` 是新的生产前端实现，需要承接原系统的业务语义、接口口径和上线能力。
- 当前 `backend/` 是最终要上线的后端代码，需要以原系统事实和现有真实路由为准，对前端提供稳定能力。

## 3. 事实源

本轮需求梳理按以下优先级认定事实：

1. `backend/app/main.py` 与已注册路由
2. `backend/app/routers/`、`backend/app/services/`、`backend/app/models/`
3. `backend/BACKEND_API_DOC.md`
4. `docs/architecture/`、`docs/decisions/`、`docs/plans/`
5. `frontend/src/api/` 与 `frontend/PROJECT.md`

## 4. 范围

本轮生产对齐至少覆盖以下模块：

- 认证与用户
- 项目列表、新建项目、全局设定
- 剧本与分集
- 主体与主体图片
- 分镜与分镜图/视频
- 创作页图片/视频/配音
- 资产库与回收站
- 剪辑成片
- API 配置、模型配置、通知与音色能力

## 5. 不做事项

- 不推翻现有前端 UI 设计
- 不擅自重构页面交互流程
- 不把联调任务扩散成全面视觉改版
- 不把原系统文档直接当成运行代码替换现有工程
- 不在未确认前重构后端核心业务模型

## 6. 核心约束

- 前端页面层不得直接请求后端，统一通过 `frontend/src/api/`
- 字段翻译、响应归一化、任务轮询、媒体 URL 拼接统一收口在前端适配层
- 后端契约以真实代码与已注册路由为准，文档需随代码回写
- 真实环境联调由人工执行，代码侧需优先保证可验证、可追踪、可回滚
- 每次任务结束都要同步回写进度文档与变更记录

## 7. 当前已识别断点

- 前端仍存在少量绕过统一请求封装的直接请求
- 部分登录/绑定相关路由与后端真实实现不完全一致
- 个别接口仍保留 stub、mock 或 no-op 承接
- 生产环境不能继续依赖 Vite 开发代理
- 媒体生成链路对 `PUBLIC_BASE_URL` 和公网可访问素材存在强依赖
- 后端个别文档描述与真实鉴权实现存在漂移

## 8. 分阶段实施

### 阶段 A：需求与事实收口

- 固化生产组合口径
- 建立原系统事实到新前端/现后端的映射表
- 明确模块范围、边界和验收清单

### 阶段 B：前端适配层收口

- 清理绕过 `request.js` 的请求
- 修正接口路径、鉴权方式、任务轮询与响应归一化
- 维持页面结构不变，仅修数据链路

### 阶段 C：后端与配置收口

- 修正阻断联调的契约漂移、环境变量和媒体访问问题
- 收口生产部署所需的 CORS、Cookie、`PUBLIC_BASE_URL`、数据库迁移说明

### 阶段 D：人工联调与文档闭环

- 由人工按模块执行真实环境联调
- 按结果补回文档、待办、风险和变更日志

## 9. 验收标准

- 新 `frontend/` 能完整承接生产后端主链路
- 不破坏现有页面 UI 和交互设计
- 所有关键模块都有明确接口契约和回写文档
- 环境变量、部署边界和媒体访问路径有清晰说明
- 后续接手者可直接从 Harness 文档快速理解当前组合关系

- 前端修复清单
- 后端/配置修复清单
- 联调验证清单
- 对应进度文档与 `CHANGELOG.md` 回写记录
