# 生成资产前端访问体验方案 B+：可执行实施清单

## 1. 文档定位

本文档是 `docs/plans/生成资产前端访问体验方案B-生产级媒体资产管线升级.md` 的执行拆分版。

主方案文档负责定义边界与目标，本清单负责把方案 B+ 拆成可排期、可验收、可回写的实施项，供后续按阶段推进。

## 2. 使用方式

后续推进方案 B+ 时，统一按以下顺序使用文档：

1. 先阅读主方案文档，确认边界、原则、目标态和验收门槛。
2. 若需要一页查看全部执行入口，优先阅读 `docs/plans/media-bplus-execution-index.md`。
3. 再按本清单拆阶段、拆任务、拆回写动作。
4. 若需要继续下钻到可直接开工的任务包，转到 `docs/plans/backlog-media-pipeline-bplus.md`。
5. 若正在推进 `MEDIA-BPLUS-10`，优先阅读 `docs/plans/MEDIA-BPLUS-10-存储分层与签名下载实现设计.md`。
6. 若正在推进 `MEDIA-BPLUS-20`，优先阅读 `docs/plans/MEDIA-BPLUS-20-媒体任务中心与状态机实现设计.md`。
7. 若正在推进 `MEDIA-BPLUS-30`，优先阅读 `docs/plans/MEDIA-BPLUS-30-视频预览与HLS交付实现设计.md`。
8. 若正在推进 `MEDIA-BPLUS-40`，优先阅读 `docs/plans/MEDIA-BPLUS-40-图片多尺寸与现代格式实现设计.md`。
9. 若正在推进 `MEDIA-BPLUS-50`，优先阅读 `docs/plans/MEDIA-BPLUS-50-观测、Runbook与灰度回滚实现设计.md`。
10. 若正在推进 `MEDIA-BPLUS-60`，优先阅读 `docs/plans/MEDIA-BPLUS-60-前端播放器与弱网降级实现设计.md`。
11. 若正在推进 `MEDIA-BPLUS-70`，优先阅读 `docs/plans/MEDIA-BPLUS-70-文档回写闭环与验收矩阵实现设计.md`。
12. 每完成一个阶段，都要回写 `docs/plans/module-progress.md`、`docs/plans/项目进度文档.md` 与 `CHANGELOG.md`。
13. 若某阶段涉及真实接口契约变更，再同步更新 `backend/BACKEND_API_DOC.md` 与 `接口变动文档.md`。

## 3. 实施总原则

- 不跳过方案 A 已建立的媒体语义分层。
- 不把“企业级”误解成“一次性做完所有基础设施”。
- 不把前端页面层改成直接猜字段或直接管理媒体策略。
- 不把媒体链路升级只做成运维问题，必须同步沉淀 Harness 文档与 Runbook。
- 真实环境联调默认由用户手工验证，我负责代码、配置、文档与验收口径收口。

## 4. 工作流拆分

方案 B+ 建议固定拆成四条并行工作流：

### 4.1 工作流 A：存储与签名访问

目标：

- 把媒体文件从“路径直出”升级到“存储分层 + 权限分层 + 受控访问”。

核心产出：

- 对象存储分桶/目录规范
- 预览与下载访问分层
- 签名 URL 策略
- 删除、失效、生命周期策略

### 4.2 工作流 B：媒体任务中心

目标：

- 把图片派生、视频转码、海报提取、HLS 切片、清理任务收口为统一的运行时链路。

核心产出：

- 任务状态机
- 幂等与重试策略
- 一致性写回规则
- 运维排障入口

### 4.3 工作流 C：前端播放与展示升级

目标：

- 在不重做现有页面视觉骨架的前提下，让前端消费真正适配企业级媒体交付。

核心产出：

- `hlsUrl / previewVideoUrl / posterUrl / downloadUrl / availableQualities` 的统一消费优先级
- 图片现代格式与渐进加载策略
- 播放失败与弱网降级策略

### 4.4 工作流 D：观测、Runbook 与 Harness 治理

目标：

- 让方案 B+ 不只是能跑，还能上线、排障、灰度、回滚、交接。

核心产出：

- 指标与 SLO
- Runbook
- 阶段验收清单
- 文档回写矩阵

## 5. 阶段化实施清单

## 5.1 阶段 0：语义与契约基线冻结

### 阶段目标

- 冻结媒体字段语义，避免进入基础设施升级后还在反复改前后端含义。

### 主要检查项

- 图片统一使用 `thumbnailUrl / previewUrl / downloadUrl`
- 视频统一使用 `posterUrl / previewVideoUrl / downloadUrl`
- 为后续 `hlsUrl / availableQualities` 预留明确字段语义
- 列表、详情、弹窗、下载共用同一来源视图模型
- 页面层不直接拼接媒体访问地址

### 重点落点

- `backend/app/services/media_view_models.py`
- `backend/app/services/media_derivative_pipeline.py`
- `frontend/src/api/`
- `docs/plans/前端展示约束规范.md`
- `docs/plans/生成资产前端访问体验方案B-生产级媒体资产管线升级.md`

### 完成标准

- 媒体字段语义有统一文档定义
- 方案 A 与方案 B+ 文档没有字段冲突
- 后续阶段不再讨论“一个 url 同时代表预览和下载”的旧口径

### 风险提示

- 若阶段 0 未冻结好字段语义，后续对象存储、签名和播放器能力上线时会反复返工

## 5.2 阶段 1：存储与访问分层

### 阶段目标

- 建立对象存储分层、预览/下载权限分层和签名访问基线。

### 任务清单

- 明确 `raw / preview / derived / hls / private-download` 的分桶或目录职责
- 明确对象 key 命名规则
- 明确公开预览、登录态预览、受控下载、内部回源四类访问等级
- 明确签名 URL 时效、作用域、单次使用限制和失效策略
- 明确删除资产后的对象清理与 CDN 失效链路

### 代码/配置落点

- `backend/app/services/` 下的媒体 URL 构造与权限校验逻辑
- 与对象存储、CDN、签名相关的配置文件或环境变量
- 后续新增的签名服务、访问控制服务

### 文档落点

- 主方案文档
- `docs/runbooks/` 下的对象存储接入与签名治理手册
- 若新增对外字段或下载口径，再更新 API 文档

### 完成标准

- 受控下载不再依赖长期裸地址
- 预览与下载的权限口径可被明确区分
- 删除链路具备“文件、缓存、审计”三层追踪能力

### 依赖关系

- 依赖阶段 0 的字段与职责基线

## 5.3 阶段 2：媒体任务中心与状态机

### 阶段目标

- 建立统一媒体运行时，让派生产物、转码、切片和清理都走 Worker 链路。

### 任务清单

- 定义任务类型：
  - `image_derivative`
  - `video_poster_extract`
  - `video_preview_transcode`
  - `video_hls_package`
  - `audio_waveform_extract`
  - `media_cleanup`
- 建立状态机：
  - `queued`
  - `running`
  - `partial_ready`
  - `retrying`
  - `ready`
  - `failed`
  - `cancelled`
- 明确幂等键与重复任务抑制
- 明确失败重试、死信、人工重驱动
- 明确“文件可读 -> 元数据回写 -> 状态切换”的一致性顺序

### 代码落点

- `backend/app/services/media_derivative_pipeline.py`
- Worker 任务定义与调度入口
- 媒体资产状态写回逻辑
- 后续如需新增任务查询接口，再同步 API 文档

### 文档落点

- 主方案文档
- 任务排障 Runbook
- 进度文档与变更日志

### 完成标准

- 不再出现“状态 ready 但产物不可访问”
- 派生产物失败有清晰可见的原因与重试动作
- Web 进程不再承担长时间媒体处理

### 依赖关系

- 依赖阶段 1 的存储层与对象地址规则

## 5.4 阶段 3：前端播放与分发体验升级

### 阶段目标

- 让前端能够稳定消费 `B+` 媒体链路，而不是继续默认直拉原始文件。

### 任务清单

- 播放器优先消费 `hlsUrl`
- `hlsUrl` 不可用时回退 `previewVideoUrl`
- 清晰度切换只依赖 `availableQualities`
- 详情默认不直拉原始大文件
- 图片列表与详情使用多尺寸和现代格式优先策略
- 弱网时默认更保守的预览策略
- 后续可扩展时间轴预览图和波形图

### 代码落点

- `frontend/src/api/`
- 相关播放器组件
- 生成资产列表、详情、预览弹窗相关页面
- 必要时补充前端媒体工具层

### 文档落点

- `docs/plans/前端展示约束规范.md`
- 主方案文档
- 前端局部 `AGENTS.md` 如需补充强约束则一并回写

### 完成标准

- 页面层依旧只消费适配层
- 视频详情不再默认把原始地址当主播放地址
- 列表、详情、下载仍保持同源媒体语义

### 依赖关系

- 依赖阶段 2 已稳定产出 `previewVideoUrl / hlsUrl / availableQualities`

## 5.5 阶段 4：观测、Runbook、灰度与上线验收

### 阶段目标

- 把方案 B+ 从“可开发”推进到“可上线、可排障、可回滚”。

### 任务清单

- 定义关键指标：
  - 派生成功率
  - 队列堆积
  - 转码平均时延
  - HLS 可用率
  - CDN 命中率
  - 下载成功率
  - 回源错误率
- 定义基础 SLO
- 建立对象存储、转码、签名、回源、CDN 故障 Runbook
- 建立灰度与回滚步骤
- 建立成本观察项：存储成本、回源成本、转码成本

### 文档落点

- `docs/runbooks/`
- `docs/plans/module-progress.md`
- `docs/plans/项目进度文档.md`
- `CHANGELOG.md`

### 完成标准

- 出现媒体访问故障时有明确排障入口
- 出现转码失败或 CDN 问题时有回滚与降级方案
- 核心指标可作为上线与扩容判断依据

### 依赖关系

- 依赖阶段 1、2、3 的主链路基本稳定

## 6. 建议的任务包拆分

为方便后续直接排期，建议将方案 B+ 固定拆成以下任务包：

### 6.1 包 A：媒体字段与契约基线

- 目标：冻结字段语义和前端消费规则
- 优先级：最高
- 预计影响：后端视图模型、前端适配层、展示规范文档

### 6.2 包 B：签名下载与访问分层

- 目标：把下载链路从长期裸地址升级为受控访问
- 优先级：高
- 预计影响：后端权限校验、签名服务、对象存储策略、下载审计

### 6.3 包 C：视频派生与 HLS

- 目标：把视频详情播放从原始大文件迁移到预览 mp4 与 HLS
- 优先级：高
- 预计影响：Worker、对象存储、播放器、详情页体验

### 6.4 包 D：图片多尺寸与现代格式

- 目标：提升列表、详情和大图查看体验
- 优先级：中
- 预计影响：图片派生、前端展示、缓存策略
- 实现设计稿：`docs/plans/MEDIA-BPLUS-40-图片多尺寸与现代格式实现设计.md`

### 6.5 包 E：观测与 Runbook

- 目标：补齐上线与排障能力
- 优先级：高
- 预计影响：监控、日志、运维手册、变更治理
- 实现设计稿：`docs/plans/MEDIA-BPLUS-50-观测、Runbook与灰度回滚实现设计.md`

### 6.6 包 F：前端播放器与弱网降级

- 目标：提升视频播放稳定性与弱网回退体验
- 优先级：中
- 预计影响：共享播放器、播放源适配、清晰度入口、观测埋点
- 实现设计稿：`docs/plans/MEDIA-BPLUS-60-前端播放器与弱网降级实现设计.md`

### 6.7 包 G：文档回写闭环与验收矩阵

- 目标：补齐阶段回写、验收、证据留存与交接治理
- 优先级：高
- 预计影响：progress、backlog、runbook、接口文档回写、阶段交接
- 实现设计稿：`docs/plans/MEDIA-BPLUS-70-文档回写闭环与验收矩阵实现设计.md`

## 7. 阶段验收清单

在任一阶段结束时，至少检查以下事项：

- 本阶段目标是否已写回主方案文档或相关专项文档
- 本阶段是否产生新的对外字段或接口契约变化
- 若有契约变化，是否已更新 `backend/BACKEND_API_DOC.md` 与 `接口变动文档.md`
- `module-progress.md` 与 `项目进度文档.md` 是否已同步
- `CHANGELOG.md` 是否已记录
- 是否形成了可复用的 Runbook 或操作规范

## 8. 当前推荐推进顺序

若后续准备正式启动方案 B+，推荐优先顺序如下：

1. 先做“字段与契约基线冻结”
2. 再做“签名下载与访问分层”
3. 再做“视频派生与 HLS”
4. 再做“前端播放器与大图策略升级”
5. 最后集中补齐“观测、成本、Runbook 与验收”

这个顺序的好处是：

- 先锁语义，避免返工
- 先控权限，避免安全债务
- 再升播放体验，避免体验优化建立在错误交付方式上
- 最后用 Harness 把上线与运维能力收口

## 9. 当前结论

方案 B+ 已经具备足够清晰的目标方案边界；接下来真正影响落地效率的，不再是“是否继续讨论媒体平台方向”，而是是否按本清单把任务拆到可以逐段执行、逐段验收、逐段回写。

因此，后续所有与生成资产企业级媒体链路相关的实施任务，都建议默认以“主方案文档 + 本实施清单”双文档作为事实源推进。
