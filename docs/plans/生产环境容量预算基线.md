# 生产环境容量预算基线

## 1. 文档定位

本文档用于把当前仓库生产运行时的默认并发参数、连接池参数和上游调用预算，收口成一份可计算的容量基线。

它解决的问题是：

- 当前默认 `WORKERS=2`、数据库池和 Redis 运行时已经存在，但没有一份统一预算口径
- 需要回答“继续加 Web worker 会不会先打爆数据库连接”这类问题
- 需要为后续压测、Supervisor/PM2 配置和告警阈值提供事实源

## 2. 当前默认配置事实

基于当前代码默认值：

- Web 进程：`backend/start_prod.sh` 默认 `WORKERS=2`
- 长任务 Worker：`backend/start_worker.sh` 当前默认启动 `1` 个独立队列 Worker 进程
- 数据库连接池：`backend/app/config.py` 默认 `DATABASE_POOL_SIZE=5`、`DATABASE_MAX_OVERFLOW=10`
- Redis：当前同时承接共享状态、最小限流和长任务队列
- 上游 HTTP client：
  - `UPSTREAM_HTTP_MAX_CONNECTIONS=100`
  - `UPSTREAM_HTTP_MAX_KEEPALIVE_CONNECTIONS=20`
  - `UPSTREAM_PROVIDER_MAX_CONCURRENCY=12`
  - `UPSTREAM_MODEL_MAX_CONCURRENCY=8`
  - `UPSTREAM_MEDIA_MAX_CONCURRENCY=16`

## 3. 数据库连接预算

### 3.1 预算公式

当前 SQLAlchemy async engine 为“每个进程一套连接池”，因此数据库最大理论占用可按下面估算：

`总连接上限 = Web 进程数 * (pool_size + max_overflow) + Worker 进程数 * (pool_size + max_overflow)`

按当前默认值：

- 每个进程数据库连接上限：`5 + 10 = 15`
- Web 进程：`2 * 15 = 30`
- Worker 进程：`1 * 15 = 15`
- 当前默认总理论上限：`45`

### 3.2 预算结论

- 如果 PostgreSQL 可用连接数低于 `60`，当前默认值已经不算宽松
- 如果后续把 Web worker 从 `2` 提到 `4`，仅 Web 侧理论上限就会变成 `60`
- 如果同时增加多个长任务 Worker，而不下调 DB pool，连接数会继续线性放大

### 3.3 当前工程约束

- 在没有压测和数据库连接数核算前，不建议把 `WORKERS` 从 `2` 直接上调到 `4`
- 若要增加 Worker 数量，优先评估是否先下调长任务 Worker 的数据库并发使用峰值
- 若生产 PostgreSQL 连接预算偏紧，下一步优先考虑：
  - 下调 `DATABASE_MAX_OVERFLOW`
  - 明确 Web/Worker 不同进程组的池配置
  - 必要时引入 `PgBouncer`

## 4. Redis 预算

当前 Redis 承接三类职责：

- 共享状态：验证码、扫码登录、手机号换绑等票据
- 限流：验证码发送、二维码轮询、任务创建/取消/重试
- 长任务队列：`BACKGROUND_JOB_QUEUE_NAME`

当前工程约束：

- 生产必须启用 Redis，`REQUIRE_REDIS_IN_PRODUCTION=true`
- 若 Redis 不可用，生产态应视为运行时故障，而不是继续依赖内存回退
- Redis 当前既是共享状态层，也是运行时控制面；后续若任务量明显上升，可再评估是否拆分逻辑库或实例

## 5. 上游 HTTP 并发预算

当前已引入统一上游 HTTP client 门面 `backend/app/services/http_client.py`，并按三类 profile 做基础限流：

- `provider`: MiniMax、常规 provider API
- `model`: LLM、图片/视频模型提交与轮询
- `media`: 外部媒体抓取、远程落盘、媒体可达性检查

默认预算：

- `provider` 并发槽位：`12`
- `model` 并发槽位：`8`
- `media` 并发槽位：`16`

解释：

- `model` 并发低于 `media`，是因为模型提交和轮询更容易被上游 429/超时放大
- `media` 并发略高，是因为远程拉取和可达性检查大多是 IO 等待，但仍需限制，避免把外部流量峰值反灌到本系统
- 连接池上限 `100` 是整个进程级 client 的硬上限，不代表所有链路都应该打满

## 6. 当前推荐生产基线

在未完成正式压测前，建议按下面口径起步：

- Web：`2` 个 worker
- 长任务 Worker：`1` 个独立 worker 进程
- PostgreSQL：至少预留 `60+` 可用连接
- Redis：独立生产实例，开启持久化与基础监控
- Nginx：承接 `/uploads/` 静态资源，不让 FastAPI 扛热点媒体下载

## 7. 下一步需要补齐的验证

- 基于真实生产 PostgreSQL `max_connections` 回填“当前可承受 Web/Worker 数量”
- 统计长任务 Worker 在任务高峰时的 DB 实际占用，而不是只看理论上限
- 为上游 HTTP client 补 429、5xx、超时的日志聚合与首轮压测记录
- 对任务列表、资产列表、分镜列表、通知列表补一轮索引和返回体审计
