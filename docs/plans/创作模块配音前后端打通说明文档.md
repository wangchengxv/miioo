# 创作模块配音前后端打通说明文档

## 1. 文档目标

本文档用于说明当前仓库中“创作模块配音功能”已经打通到什么程度、前后端之间的真实调用关系是什么、能力层分别由谁负责，以及目前还剩哪些未彻底闭环的点。

本文档只描述当前仓库里的真实代码事实，不把历史设想、旧文档口径或 UI 表象直接当成已生效能力。

---

## 2. 一句话结论

当前“创作页配音”主链路已经完成以下闭环：

- 创作页按 `voice` 分类读取可用配音模型
- 当前所选配音模型可以反向决定“模型官方音色库”来源
- 用户可在共享音色弹窗中选择 `模型官方音色 / miioo 音色库 / 自定义音色`
- 用户可提交文本、语速、情绪、音色、模型到后端生成配音
- 后端可基于用户当前启用的 voice 模型执行 TTS，并将结果落为 `AudioClip + Asset`
- 前端可回显创作配音历史，并继续执行播放、收藏、删除、下载等动作

但当前仍有两个重要边界需要明确：

- 长文本异步配音能力后端已经提供 `POST /api/creation/audios/generate-async` 与 `GET /api/creation/audios/tasks/{task_id}`，前端创作页主提交流程当前已可按既有统一入口自动切到异步链路
- 创作页现已把提示词里的 `@` 数字资产绑定正式复用到配音生成：前端会提交 `prompt_raw / prompt_resolved / mentions / attachments`，后端会把绑定关系写入音频资产 metadata，并按“显式参考音频 > 被 `@` 的音频资产 > 其它音频附件”的优先级确定 `reference_audio_url`

## 4. 需求分析

### 4.1 业务目标

创作模块里的配音能力，本质上要解决的是下面这条业务链：

1. 用户在创作页切到“配音”模式
2. 用户选择一个可用配音模型
3. 用户选择一个音色来源
4. 用户输入文本，并设置语速、情绪等参数
5. 系统调用后端真实 voice 模型执行 TTS
6. 结果回写为可播放、可收藏、可下载、可回看历史的创作音频资产

### 4.2 前端侧约束

- 页面层不能直接请求后端，必须统一走 `frontend/src/api/`
- 页面层不应自己翻译 provider、voice、media 字段语义
- 音色来源差异必须收口在共享弹窗和 API 适配层，而不是散落到多个页面
- 配音结果必须和创作页其它资产一样，统一进入创作历史与资产语义体系

### 4.3 后端侧约束

- 模型选择必须受用户当前已启用的 `voice` 模型控制
- 音色 ID 不能直接裸透传给上游，必须先经过音色上下文解析
- 上游返回的外部音频地址需要先落盘，再入库为系统自己的媒体资产
- 长文本异步能力与短文本同步能力需要明确分口径，不应混用

## 6. 前端真实调用关系

### 6.1 页面入口

创作模块的配音入口在 `CreationPage.jsx` 内部，通过 `genType === 'dubbing'` 切换到配音模式。页面会把该模式视为独立于图片、视频的第三类创作能力。

### 6.2 模型选择

配音模式下，前端按 `voice` 分类读取模型列表，并把当前选中的配音模型保存为创作输入的一部分。

这一步的关键意义不是只展示一个下拉框，而是建立下面这条能力映射：

- 当前模型属于哪个 provider
- 当前 provider 对应哪个“模型官方音色库”
- 当前用户到底有没有可用 voice 模型

也就是说，创作页不是直接“先选音色再猜模型”，而是先有模型，再根据模型反推出官方音色来源。

### 6.3 音色弹窗

点击配音输入区左侧上传占位卡，实际打开的是共享组件 `DubbingVoiceModal.jsx`。

弹窗内部把音色来源拆成四类：

- `model`：模型官方音色库
- `miioo`：miioo 系统音色库
- `custom`：用户自定义音色
- `fav`：历史音频作品

它们对应的前端接口分别是：

- `GET /api/voices/official`
- `GET /api/voices/library?provider=miioo`
- `GET /api/voices?tab=custom`
- `GET /api/creation/audios`

弹窗确认后，会把统一结构回传给 `CreationPage.jsx`：

```json
{
  "id": "voice_id_or_audio_id",
  "name": "展示名",
  "source": "aiping|minimax|volcengine|miioo|custom|fav",
  "referenceAudioUrl": "可试听或可复用的音频地址"
}
```

这里最关键的前端收口点有两个：

- 页面层不直接关心后端原始 voice 字段差异，而是只接收弹窗确认后的统一视图模型
- “当前选中模型”会影响 `model` 页签默认读取哪个 provider 的官方音色目录

### 6.4 提交生成

用户点击发送后，`CreationPage.jsx` 会从输入区提取以下信息：

- `prompt`，即要配音的文本
- `model`
- `speed`
- `emotion`
- `voice_id`
- `reference_audio_url`
- `prompt_raw / prompt_resolved`
- `mentions / attachments`
- `session_id / shot_id / project_id`

然后调用 `src/api/creation.js` 的 `apiGenerateCreation(params)`。

配音分支下的前端适配逻辑是：

1. 若带了本地音频文件，先调用 `apiUploadCreationAudio(...)` 上传参考音频
2. 统一提取 `prompt_raw / prompt_resolved / mentions / attachments`
3. 将上传结果、被 `@` 的音频资产或已存在的 `reference_audio_url` 写回请求体
4. 调用 `apiGenerateCreationAudio(...)`
5. 从返回值中抽取 `audio_url / id / asset_id`
6. 将结果归一为创作页结果卡需要的 `audioItems`

### 6.5 结果承接

同步配音成功后，`CreationPage.jsx` 会把结果写入 `creationStore.js` 的 `generationsByTab.dubbing`。

刷新或再次进入创作页时，页面还会通过 `apiListCreationAudiosPage(...)` 读取历史音频列表，并与本地结果合并，保证“刚生成的结果”和“历史已生成结果”使用同一套展示模型。

因此前端当前已经形成了：

- 本次生成即时回显
- 历史分页回显
- 收藏/删除/批量删除
- 下载
- 音频播放

这几个基本闭环。

## 7. 后端真实调用关系

### 7.1 主入口

创作页同步配音的主入口是：

- `POST /api/creation/audios/generate`

长文本异步入口是：

- `POST /api/creation/audios/generate-async`

异步轮询入口是：

- `GET /api/creation/audios/tasks/{task_id}`

### 7.2 同步配音执行链

同步配音的真实执行顺序如下：

1. `creation.py` 读取 `CreationAudioGenerateRequest`
2. 先校验文本不能为空，且长度不能超过同步上限
3. 通过 `_resolve_creation_scope(...)` 解析 `session / shot / project`
4. 通过 `get_user_model_provider_credentials(...)` 选择当前用户可用的 `voice` 模型
5. 通过 `_resolve_creation_audio_voice_context(...)` 解析前端传来的 `voice_id`
6. 通过 `build_tts_provider_options(...)` 组装语速、情绪和高级 TTS 参数
7. 调用 `tts_service.generate(...)` 真正访问上游 TTS provider
8. 通过 `persist_if_external(...)` 将外部音频地址落盘到本地上传目录
9. 通过 `_create_creation_audio_records(...)` 写入 `AudioClip + Asset`
10. 返回前端可直接消费的创作音频结果

### 7.3 音色上下文解析

`audio_voice_context.py` 负责把前端传进来的 `voice_id` 解析成真正可传给上游的 `upstream_voice_id`。

这一步是当前配音能力层打通的关键，因为前端选择到的音色并不总能直接等于厂商真实 voice id：

- 自定义音色需要映射到 `provider_voice_id`
- miioo 系统音色有一层兼容映射
- 自定义音色还要校验 clone 状态是否 `ready`

因此，当前真正生效的不是“前端选什么就原样发什么”，而是：

- 前端负责选择一个统一音色标识
- 后端负责把它还原成上游真正需要的 voice id

### 7.4 TTS 执行层

`tts_service.generate(...)` 是统一 TTS 门面，会根据 `provider_type` 分流到不同 provider 的运行时实现。

这意味着创作页配音在业务层看起来只有一个“生成配音”按钮，但后端实际已经做了 provider 级收口：

- 模型选择决定 provider
- provider 决定调用哪个运行时
- 运行时统一回传 `url / duration / metadata`

页面层并不直接感知这些分支。

### 7.5 结果落库层

同步生成成功后，后端会同时写入两类数据：

- `AudioClip`
- `Asset`

它们的职责不同：

- `AudioClip` 更偏业务记录，保存文本、音色、语速、情绪、音频地址
- `Asset` 更偏统一资产承接，保存名称、模型、`metadata_json`、媒体语义

`metadata_json` 里会保留：

- `source = creation_audio`
- `session_id / shot_id / project_id`
- `model / provider_type`
- `voice_id / voice_name / provider_voice_id`
- `speed / emotion`
- `tts_metadata / tts_request_options`

这也是后续历史回显、下载、能力追踪和问题排查的事实源。

## 9. 已打通的核心闭环

当前可以明确认为已经打通的内容如下：

### 9.1 模型到音色来源的打通

创作页当前选中的配音模型，已经能真实影响共享音色弹窗中的官方音色来源，不再是 UI 假切换。

### 9.2 音色选择到后端执行的打通

用户在弹窗里确认的音色，已经可以通过 `voice_id -> voice_context -> upstream_voice_id` 这条链路影响后端真实 TTS 调用。

### 9.3 配音结果落库的打通

生成结果不会只停留在临时返回值，而是会被落为 `AudioClip + Asset`，并重新进入创作历史体系。

### 9.4 历史结果回显的打通

创作页结果区和音色弹窗“音频作品”页签，当前都已经可以消费真实的创作音频历史，而不是只看本地临时状态。

## 10. 当前未完全闭环的点

### 10.1 长文本异步配音未接到创作页主提交流程

后端已经提供：

- `POST /api/creation/audios/generate-async`
- `GET /api/creation/audios/tasks/{task_id}`

而前端 `src/api/creation.js` 当前配音分支仍固定走：

- `POST /api/creation/audios/generate`

这意味着：

- 短文本配音主链路已可用
- 长文本异步能力后端有，前端主入口还没真正切过去

### 10.2 `@` 资产绑定在配音链路中的当前语义

前端当前允许用户在配音输入区通过 `@` 绑定图片、视频、音频资产。

当前已经稳定的执行语义是：

- `@资产` 只做参考绑定，不进入最终送给 TTS 的正文
- 若存在显式 `reference_audio_url`，优先使用显式值
- 若没有显式参考音频，但存在被 `@` 的音频资产，则优先使用该音频资产 URL
- 若两者都没有，再回退到普通音频附件
- `@图片 / @视频` 不直接影响 TTS 执行参数，但会随 `mentions / asset_bindings` 一起写入音频资产 metadata，供历史回显与排障使用

### 10.3 前端对异步与同步口径的选择策略尚未收口

当前后端同步接口有文本长度限制，异步接口也已经存在，但前端还没有形成类似下面的统一策略：

- 短文本默认走同步
- 超过阈值自动切异步
- 或者特定 provider 统一走异步

这部分仍属于后续收口项。

## 11. 推荐的后续打通顺序

若后续要继续把创作模块配音做成完全闭环，建议按下面顺序推进。

### 11.1 第一优先级

把创作页前端主提交流程接到长文本异步配音：

- 前端在 `apiGenerateCreation(...)` 的配音分支中增加同步/异步判断
- 当文本长度超过同步阈值，直接调用 `POST /api/creation/audios/generate-async`
- 轮询 `GET /api/creation/audios/tasks/{task_id}`
- 任务完成后再统一落回当前结果卡和历史列表刷新逻辑

### 11.2 第二优先级

继续观察不同 provider 对参考音频语义的实际表现：

- 当前同步与异步配音入口都已经正式支持 `reference_audio_url`
- 当前已将“是否使用参考音频、绑定了哪些资产”写入 `metadata_json`
- 若后续某个 provider 需要更细粒度的 reference voice 专属字段，再在现有请求体基础上做兼容增强

### 11.3 第三优先级

把配音能力的“页面层展示”与“执行层事实”继续对齐：

- 明确哪些 provider 真支持参考音频
- 明确哪些 provider 只支持 `voice_id`
- 将这类差异继续收口在后端能力层和前端适配层，而不是让页面自己猜

---

## 12. 结论

当前创作模块的配音功能，已经完成了“模型可选、音色可选、短文本可生、结果可落库、历史可回显”的主闭环，属于真实可用状态，不再是单纯 UI 占位。

但如果从“完全生产化”的标准看，仍有两段链路需要继续补齐：

- 长文本异步配音还没有被前端创作主入口正式消费
- 参考音频虽然已经进入前端选择与参数预留层，但还没有完全进入后端同步配音执行事实层

因此当前最准确的判断应当是：

- 创作模块配音主链路已打通
- 能力层拆分已经基本成型
- 异步与参考音频两条增强链路还处在“后端已具备或前端已预留，但未完全闭环”的阶段
