# 热点查询索引审计

## 1. 目标

本审计用于补齐 `P1` 中“热点列表与聚合查询索引补审”的事实留仓。

范围聚焦当前最可能在生产高并发下放大的读路径：

- 任务列表
- 资产列表
- 分镜列表
- 项目列表与项目概览
- 通知列表/未读数
- 模型默认配置读取

## 2. 已确认的现状

### 2.1 已有较匹配的索引

- `projects`
  - 已有 `ix_projects_user_updated_id_desc`
  - 适配项目列表 `user_id + updated_at desc + id desc`
- `storyboards`
  - 已有 `ix_storyboards_project_sort_shot_created`
  - 已有 `ix_storyboards_project_episode_sort_shot_created`
  - 适配项目级分镜列表和分集分镜列表
- `gen_tasks`
  - 已有 `ix_gen_tasks_user_created_id_desc`
  - 适配默认任务列表倒序

### 2.2 已确认的主要风险

- `notifications`
  - 列表、未读数、全部已读都依赖 `user_id` 单列索引
  - 缺少与 `is_read + created_at` 对齐的复合索引
- `gen_tasks`
  - `status` / `project_id` 筛选没有复合索引承接
- `assets`
  - “项目下全资产列表”缺少 `user_id + project_id + is_deleted + created_at` 这条主索引
  - 关键词搜索仍是 `ILIKE %keyword%`，当前不适合靠 btree 解决
- `projects/{id}/overview`
  - 一次请求中包含多次 count/list + Python 聚合
  - `subjects(project_id, type)`、`compositions(project_id, user_id, status)` 之前没有对应复合索引
- `model_configs`
  - 高频默认模型选择走 `user_id + category + is_enabled + is_default`
  - 之前只有 `user_id` 单列索引

## 4. 当前仍未收口的风险

### 4.1 仍需后续索引或搜索增强

- 真实环境迁移执行后的索引命中率与体积观测
- 是否还需要为更长文本路径引入全文检索

这些路径如果后续成为高频入口，应考虑：

- `pg_trgm`
- `GIN`
- 或全文检索

### 4.2 仍需分页治理

- 多个接口仍是 `offset/limit`，深分页成本会继续上升
- `assets` 已具备兼容式 cursor/keyset 能力，但当前页面层尚未正式切到该口径
