# 总结成功经验

## 1. 场景

本次成功经验来自一次真实云端部署排障：

- 线上页面大量图片破图
- 浏览器控制台持续出现 `/uploads/...` 相关 `404`
- 资源文件在服务器磁盘上真实存在
- 前端域名、后端域名、Nginx 配置、后端 `.env` 多处因素交织，容易误判

最终确认问题已修复，线上 `curl -I` 已返回：

- `HTTP/2 200`
- `content-type: image/png`
- `x-static-handled: nginx`

这说明 `/uploads/...` 已经由 Nginx 静态直出，而不再误落到 FastAPI 的 `404 JSON`。

## 2. 最终结论

本次问题的真实根因不是前端组件，也不是数据库字段本身，而是：

1. 线上 `/uploads/...` 静态文件原本没有真正命中 Nginx 静态规则
2. Nginx 配置虽然已经改对，但运行中的线上服务在排障前并未真正 reload 到最新配置
3. 后端 `.env` 中还存在 URL 配置写法不规范的问题，增加了排障噪音

一句话总结：

**文件存在 + 路径正确 + 配置文件看起来正确，不代表线上请求已经命中正确规则；真实线上结果必须用 `curl` 和响应头验证。**

## 3. 本次有效的排障顺序

这次最有效的地方，不是一次性猜对根因，而是排障顺序正确。

### 3.1 先看浏览器真实请求 URL

不要只看控制台里简写的文件名。

正确做法：

- 打开浏览器 `Network`
- 找到破图请求
- 复制完整 `Request URL`

本次就是通过这一步先排除了“前端域名拼错”的可能，确认请求已经打到了真实后端域名 `https://www.miiooai.com/uploads/...`

### 3.2 再确认文件是否真实存在于磁盘

这一步非常关键，能快速区分“路径错”还是“文件丢”。

本次有效命令：

```bash
find /www/wwwroot/miiooaib.com/backend/uploads -name "e0315ca260024d1c84cae2fdfcc05a0d.png"
ls -lah /www/wwwroot/miiooaib.com/backend/uploads/creation/sessions/<session_id>/shots/<shot_id>/images/
```

本次结论：

- 文件真实存在
- 文件路径与请求 URL 对应得上

这一步直接排除了“上传失败”“文件没落盘”“目录不存在”三类常见误判。

### 3.3 再用 `curl -I` 验证公网响应

只看浏览器还不够，必须直接测线上 URL。

本次关键命令：

```bash
curl -I https://www.miiooai.com/uploads/creation/sessions/<session_id>/shots/<shot_id>/images/<filename>.png
```

排障前返回：

- `404`
- `content-type: application/json`

这条结果非常关键，因为它说明请求并没有走 Nginx 静态目录，而是落到了应用层。

排障后返回：

- `200`
- `content-type: image/png`
- `x-static-handled: nginx`

这一步最终坐实问题已修复。

### 3.4 结合响应头判断究竟是谁在处理请求

这是本次排障中最有价值的经验之一。

如果返回：

- `content-type: application/json`

优先怀疑：

- 请求被转发给 FastAPI
- 不是 Nginx 在直出静态文件

如果返回：

- `content-type: image/png`
- `x-static-handled: nginx`

说明：

- `/uploads/...` 已经被 Nginx 正确接管

本次正是通过这个判断，把问题从“文件是不是存在”收口到了“线上实际命中的处理链路是谁”。

## 4. 本次确认有效的配置口径

### 4.1 后端 `.env` 需要显式写死上传目录

后端 `.env` 应明确配置：

```env
PUBLIC_BASE_URL=https://www.miiooai.com
UPLOAD_DIR=/www/wwwroot/miiooaib.com/backend/uploads
SERVE_UPLOADS_VIA_APP=false
```

经验结论：

- 生产环境不要依赖 `UPLOAD_DIR` 默认值
- 要让后端写盘目录与 Nginx `alias` 指向完全一致

### 4.2 `.env` 里 URL 不要带反引号和多余空格

错误写法：

```env
PUBLIC_BASE_URL= `https://www.miiooai.com`
ONELINKAI_BASE_URL= `https://api.onelinkai.cloud`
```

正确写法：

```env
PUBLIC_BASE_URL=https://www.miiooai.com
ONELINKAI_BASE_URL=https://api.onelinkai.cloud
```

经验结论：

- `.env` 值不要带反引号
- `=` 后不要保留无意义空格
- 否则会让“环境变量是不是已经正确生效”变得难以判断

### 4.3 Nginx 必须明确接住 `/uploads/`

本次已验证有效的核心规则是：

```nginx
location ^~ /uploads/ {
    alias /www/wwwroot/miiooaib.com/backend/uploads/;
    add_header Access-Control-Allow-Origin *;
    add_header Cache-Control "public, max-age=604800, immutable";
    add_header X-Static-Handled "nginx" always;
    expires 7d;
    access_log off;
    autoindex off;
}
```

经验结论：

- `alias` 末尾斜杠不能省
- 要优先于 `location /` 生效
- 最好保留 `X-Static-Handled: nginx` 作为排障响应头

### 4.4 改完 Nginx 后必须真的 reload

这是本次排障里最容易踩坑的一点。

问题表现：

- 配置文件内容已经改对
- `nginx -T` 也能看到新配置
- 但浏览器请求仍旧像在走旧逻辑

最终经验：

```bash
nginx -t && nginx -s reload
```

不要只看“文件内容已改”，一定要再用真实 URL 做 `curl -I` 验证。

## 5. 本次最值得复用的判断方法

以后再遇到“线上图片 / 视频 / 音频破图或 404”，优先按这个顺序排：

1. 浏览器 `Network` 复制完整 `Request URL`
2. 服务器 `find` / `ls` 验证文件是否真实存在
3. `curl -I` 验证公网响应头和状态码
4. 用 `content-type` 与 `X-Static-Handled` 判断是 Nginx 还是应用层在处理
5. 再回头查 Nginx `alias`、后端 `.env`、前端 `VITE_API_BASE_URL`

这条顺序比“先猜是前端问题、先猜是数据库问题、先猜是 Nginx 配置错了”更稳定。

## 6. 这次排障后沉淀出的长期规则

### 6.1 生产静态资源问题必须以真实线上响应为准

不要只信：

- 代码仓里的配置文件
- 本地推测
- 控制台里被折叠的文件名

必须信：

- 线上 `curl -I`
- 浏览器完整 `Request URL`
- 服务器磁盘真实文件路径

### 6.2 `/uploads` 问题优先分三层看

分层顺序：

1. 浏览器请求的是哪个域名、哪个路径
2. 服务器磁盘上有没有这个文件
3. Nginx 是否把这条路径按静态文件处理

这三层一旦分开，排障会快很多。

### 6.3 为静态规则保留排障响应头是值得的

本次 `X-Static-Handled: nginx` 非常有用。

经验结论：

- 生产静态文件规则建议保留一个明确响应头
- 这样以后能一眼看出请求是否真的命中了 Nginx 静态层

## 7. 本次可直接复用的命令

```bash
# 1. 查真实请求文件是否存在
find /www/wwwroot/miiooaib.com/backend/uploads -name "<filename>"

# 2. 看目标目录
ls -lah /www/wwwroot/miiooaib.com/backend/uploads/creation/sessions/<session_id>/shots/<shot_id>/images/

# 3. 看线上返回的是静态文件还是应用层 JSON
curl -I https://www.miiooai.com/uploads/creation/sessions/<session_id>/shots/<shot_id>/images/<filename>

# 4. 校验并重载 Nginx
nginx -t && nginx -s reload

# 5. 看当前生效配置里是否真的有 /uploads 规则
nginx -T | grep -n "/uploads/" -A 6 -B 6
```

## 8. 最终经验总结

这次最成功的不是“碰巧把问题修好了”，而是验证链路完整：

- 先确认请求 URL
- 再确认文件存在
- 再确认公网响应
- 最后通过响应头证明处理链路已经切回 Nginx 静态层

因此，这次经验可以沉淀成一句长期有效的话：

**生产静态资源排障，不要只看配置文件长什么样，要看线上请求最后是谁处理了它。**
