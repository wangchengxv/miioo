# 生成资产前端访问体验方案 A：轻量媒体分层交付

## 1. 文档定位

本文档用于沉淀当前 miioo 项目在“生成资产文件过大、前端访问慢、预览和播放体验差”问题下的近期主路线方案。

本方案是当前阶段的正式推荐方案，目标是在尽量不改动整体基础设施的前提下，优先解决最明显的用户体验问题。

## 2. 为什么当前优先选方案 A

结合当前仓库代码事实，方案 A 最适合现在推进，原因如下：

- 前端已经具备媒体语义收口基础，页面正在逐步消费 `imageUrl / thumbnailUrl / videoUrl / audioUrl / downloadUrl` 等明确字段。
- 后端已经具备部分缩略图派生能力，并不是从零开始。
- 生产治理方向已经明确 `/uploads` 应由 `Nginx` 承接静态直出，当前适合先做“媒体分层交付”，而不是直接全面升级基础设施。
- 当前主线仍是“真实环境联调验收 + 长任务稳定性收口 + 生产性能治理收口”，不适合把任务扩散成一次完整的媒体平台重构。

一句话讲，当前最该优先解决的是：

`不要再让前端列表、弹窗和默认播放直接吃原始大文件。`

## 3. 核心目标

在不大改基础设施的前提下，把媒体交付拆层：

- 列表只拿轻预览
- 详情默认拿中清预览
- 播放默认拿轻量可播版本
- 下载才拿原始文件

## 4. 技术路线

### 4.1 图片资产三层化

所有图片类资产统一拆成三类地址：

- `thumbnailUrl`：用于列表卡片、宫格、历史记录右栏
- `previewUrl`：用于详情弹窗、资产预览、编辑回显
- `downloadUrl`：用于原图下载或“查看原图”

建议控制口径：

- `thumbnailUrl`：`240 ~ 480px`
- `previewUrl`：`1280 ~ 1920px`
- `downloadUrl`：原始结果或最高质量托管版本

### 4.2 视频资产四层化

所有视频类资产统一拆成四类地址：

- `posterUrl`：用于列表与详情首帧图
- `previewVideoUrl`：用于详情默认播放的压缩预览版 mp4
- `videoUrl`：当前业务主播放地址，短期可与 `previewVideoUrl` 同义
- `downloadUrl`：用于原始视频下载

当前阶段不强制要求 HLS，但必须先做到：

`预览播放地址和原始下载地址彻底分开。`

### 4.3 列表接口轻量化

列表接口只返回首屏必需字段：

- `id`
- `assetType`
- `thumbnailUrl / posterUrl`
- `duration`
- `width / height`
- `sizeBytes`
- `status`
- `createdAt`

列表接口默认不再返回：

- 原始大文件 URL
- 全量生成参数
- 全量参考素材
- 大体积 metadata
- 历史版本全集

### 4.4 详情接口承接重字段

详情接口再补充：

- `previewUrl / previewVideoUrl`
- `downloadUrl`
- `imageUrl / videoUrl / audioUrl`
- 完整生成参数
- 参考素材
- 业务元数据

这样用户的感知会从“页面整体很慢”变成“页面先开，详情再补齐”。

### 4.5 前端交互策略

前端默认交互建议：

- 卡片先展示 `thumbnailUrl / posterUrl`
- hover 时再轻量预取详情预览资源
- 弹窗打开先秒出 `previewUrl / posterUrl`
- 视频默认 `preload="metadata"`
- 用户明确点击“下载原文件”时才访问 `downloadUrl`

### 4.6 后端后处理任务化

当前生成任务完成后，不应只产出一个原始文件，而应追加轻量后处理：

- 生成缩略图
- 生成海报图
- 生成预览版图片
- 生成预览版视频
- 写入衍生状态字段

建议增加统一状态：

- `derivativeStatus`
- `previewReady`
- `previewFailedReason`

让页面能够明确展示：

- 预览版准备中
- 预览版可用
- 原文件可下载

## 5. 适用范围

本方案优先适用于以下模块：

- 资产库
- 主体页历史图与详情弹窗
- 分镜页历史图与视频预览
- 创作页图片/视频结果区
- 剪辑成片素材池

## 6. 方案收益

本方案最直接的收益：

- 列表打开更快
- 弹窗预览更快
- 视频首帧更快
- 黑屏等待更少
- 下载和预览职责更清晰
- 前端懒加载终于建立在正确的媒体分层前提上

## 7. 方案边界

本方案不能彻底解决的问题：

- 弱网下高清长视频播放卡顿
- 大量外部分享访问的带宽压力
- 视频 seek 拖动的最佳体验
- 大规模媒体分发与跨区域访问

这些问题属于方案 B 的承接范围。

## 8. 推荐实施顺序

### 阶段 1：字段分层

- 图片统一补齐 `thumbnailUrl / previewUrl / downloadUrl`
- 视频统一补齐 `posterUrl / previewVideoUrl / downloadUrl`
- 适配层统一回退顺序

### 阶段 2：接口分层

- 列表接口轻量化
- 详情接口重字段下沉
- 页面默认只消费轻预览

### 阶段 3：后处理分层

- 生成后追加派生产物任务
- 页面展示预览准备状态
- 下载入口与预览入口彻底分离

## 9. 当前结论

方案 A 是当前 miioo 项目最适合先落地的方案。

它不是终局方案，但它能在最小改动下最快解决“生成资产太大、前端访问慢、懒加载也不够用”的当前核心问题。
