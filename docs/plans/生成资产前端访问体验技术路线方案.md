# 生成资产前端访问体验技术路线方案

## 1. 文档定位

本文档用于针对当前 miioo 项目“生成资产文件过大、前端访问慢、预览/播放体验差”的问题，选出最适合现阶段工程现实的技术路线，并给出两套可执行方案：

- 方案 A：基于现有工程快速落地的轻量媒体分层交付路线
- 方案 B：面向生产规模与更高体验上限的媒体资产管线升级路线

本文档负责：

- 基于当前仓库代码事实选择适合本项目的路线，而不是给通用空方案
- 说明每套方案解决什么问题、依赖什么基础、会新增什么复杂度
- 给出前端适配层、后端产物层、部署层和交互层的统一落地建议

本文档不负责：

- 直接替代后续详细接口设计
- 直接替代对象存储、CDN、转码服务的具体部署手册
- 直接替代真实环境压测和成本测算结论

## 2. 当前项目代码事实

基于当前仓库，和本问题直接相关的事实如下：

- 前端已经在 `frontend/src/api/` 与 `frontend/src/utils/mediaItem.js` 建立了媒体语义收口基础，页面层正在逐步改为只消费 `imageUrl / thumbnailUrl / videoUrl / audioUrl / downloadUrl` 等明确字段。
- 后端已经在主体、分镜、创作、工作台等链路中具备“缩略图派生”的初步能力，说明当前工程并不是完全没有媒体分层基础。
- 生产运行治理文档已经明确把 `/uploads` 静态流量剥离到 `Nginx` 直出作为既定方向，这意味着“应用层不直接扛大媒体流量”已经是当前项目既定口径。
- 当前主线仍处于“真实环境联调验收 + 长任务稳定性收口 + 生产性能治理计划收口”阶段，而不是完全空白的新基建阶段。
- 当前视频链路虽然已有缩略图、首帧和任务制承接，但尚未形成真正的“预览版视频 / 自适应码率 / 原始下载”三层交付体系。
- 当前图片、视频、音频大量仍以本地 `/uploads` 文件直出为主，尚未统一进入对象存储 + CDN + 转码切片体系。

基于这些事实，当前最重要的决策不是“要不要优化”，而是“优先做低风险高收益的分层交付，还是直接引入完整媒体基础设施”。

## 3. 选型结论

### 3.1 总结论

当前项目最适合采用“双路线并行、分阶段推进”的策略：

1. 近期主路线选择方案 A，作为当前生产体验优化的主落地路线。
2. 中期演进目标选择方案 B，作为后续媒体规模、外部分享和高清播放需求上来后的升级路线。

### 3.2 为什么不建议现在一步到位只做方案 B

原因不是方案 B 不好，而是它对当前项目阶段来说改动面太大：

- 会同时引入对象存储、CDN、异步转码、任务编排、回源策略和播放协议变更。
- 会把当前“联调和主链路稳定性”任务扩散成一次基础设施升级项目。
- 会显著增加后端部署、运维和排障复杂度。
- 若当前媒体访问瓶颈主要集中在“列表误拉大文件、详情直接吃原文件、视频没有预览层”，则先做方案 A 往往就能解决 `60% ~ 80%` 的用户感知问题。

### 3.3 为什么不能只停留在方案 A

因为当前项目本质上是 AIGC 影视化工作流，天然会继续增长：

- 视频资产会越来越多
- 单文件尺寸会越来越大
- 外部模型生成的媒体格式会越来越杂
- 用户会更频繁地进行预览、回看、下载和分享

当这些需求继续增长时，仅靠本地 `/uploads + Nginx + 缩略图` 会逐渐触顶，所以方案 A 应视为“当前阶段最优路线”，方案 B 应视为“明确的下一阶段目标”。

## 4. 方案 A：轻量媒体分层交付路线

### 4.1 适用阶段

适用于当前项目的近期主线：

- 仍在持续真实环境联调
- 不希望大改现有部署结构
- 已有 `/uploads + Nginx` 静态直出基础
- 已有缩略图派生和媒体语义字段收口基础
- 希望用最小改动优先提升列表、弹窗、详情和播放体验

### 4.2 核心目标

不改动整体基础设施前提下，先把“前端访问的到底是什么文件”分层：

- 列表只拿轻预览
- 详情默认拿中清预览
- 下载才拿原始文件
- 视频播放优先拿轻量可播版本和海报图

### 4.3 技术路线

#### A-1. 图片资产三层化

所有图片类资产统一拆分为：

- `thumbnailUrl`：列表卡片、宫格、历史记录右栏
- `previewUrl`：详情弹窗、资产预览、编辑回显
- `downloadUrl`：原图下载或“查看原图”

建议口径：

- `thumbnailUrl` 控制在 `240 ~ 480px` 宽度
- `previewUrl` 控制在 `1280 ~ 1920px` 边界
- `downloadUrl` 保留原始生成结果或最高质量托管版本

#### A-2. 视频资产四层化

所有视频类资产统一拆分为：

- `posterUrl`：列表与详情首帧图
- `previewVideoUrl`：详情默认播放的压缩预览版 mp4
- `videoUrl`：当前业务主播放地址，短期内可先与 `previewVideoUrl` 同义
- `downloadUrl`：原始视频下载地址

短期内不强制要求 HLS，但必须先明确“预览播放”和“原始下载”不能继续共用同一个宽泛 `url`。

#### A-3. 列表接口轻量化

列表接口只返回首屏必须字段：

- `id`
- `assetType`
- `thumbnailUrl / posterUrl`
- `duration`
- `width / height`
- `sizeBytes`
- `status`
- `createdAt`

不要在列表接口里默认带：

- 原始大文件 URL
- 全量生成参数
- 全量参考素材
- 大体积 metadata
- 历史版本全集

#### A-4. 详情接口再拉重字段

详情接口再补：

- `previewUrl / previewVideoUrl`
- `downloadUrl`
- 真实 `imageUrl / videoUrl / audioUrl`
- 完整生成参数
- 参考素材
- 业务元数据

这样用户第一感知会从“页面整体很慢”变成“页面先开，点进详情时再补内容”。

#### A-5. 前端交互策略

前端默认策略建议统一为：

- 卡片先展示 `thumbnailUrl / posterUrl`
- hover 时再轻量预取详情预览资源
- 弹窗打开先秒出 `previewUrl / posterUrl`
- 视频默认 `preload="metadata"`，避免一打开就全量拉流
- 用户明确点“下载原文件”时才访问 `downloadUrl`

#### A-6. 后端后处理任务化

当前生成任务完成后，不应只产出一个原始文件，而应追加轻量后处理：

- 生成缩略图
- 生成海报图
- 生成预览版图片或预览版视频
- 写入衍生状态字段

建议增加统一状态：

- `derivativeStatus`
- `previewReady`
- `previewFailedReason`

这样页面可明确展示“预览版准备中 / 已可预览 / 原文件可下载”。

### 4.4 适合当前项目的原因

这是当前项目近期最合适的主路线，原因如下：

- 能最大化复用当前已存在的媒体语义收口和缩略图派生能力。
- 能延续当前 `/uploads` 由 `Nginx` 直出的部署口径，不需要立刻上对象存储。
- 不会破坏当前页面视觉和交互骨架，只改媒体交付语义。
- 能直接改善资产库、主体页、分镜页、创作页和剪辑素材池的首屏与回看体验。
- 工程复杂度远低于完整的转码切片和对象存储方案，更适合当前“主链路稳定性优先”的阶段。

### 4.5 方案 A 的主要收益

- 列表打开更快
- 弹窗预览更快
- 视频黑屏和首帧慢问题明显减少
- 下载链路和预览链路职责更清晰
- 前端懒加载不再建立在“大文件仍要直接拉”的错误前提上

### 4.6 方案 A 的边界

方案 A 不能彻底解决的问题：

- 弱网下高清视频长时间播放卡顿
- 大量外部分享访问时的回源与带宽成本
- 视频拖动 seek 的最佳体验
- 大规模媒体分发和跨区域访问

这些问题需要方案 B 承接。

## 5. 方案 B：生产级媒体资产管线升级路线

补充说明：

- `2026-06-18` 起，方案 B 已进一步细化并升级为 `B+ 渐进增强` 企业级版本，详见 `docs/plans/生成资产前端访问体验方案B-生产级媒体资产管线升级.md`。
- 本总文档仍保留“方案 A / 方案 B”的路线级对比职责；更细的安全、稳定性、Harness 治理、阶段化落地与验收门槛，以独立方案 B+ 文档为准。

### 5.1 适用阶段

适用于以下阶段：

- 当前主链路和媒体语义已经稳定
- 真实环境用户量、媒体量和播放量明显增长
- 需要更高质量播放、更快拖动、更稳分享和更低应用层带宽成本
- 可以接受新增对象存储、CDN 和转码任务基础设施

### 5.2 核心目标

把“媒体文件”从应用层本地文件，升级为“可分发、可转码、可缓存、可多质量交付”的媒体资产管线。

### 5.3 技术路线

#### B-1. 存储层升级

媒体主存储从本地 `/uploads` 渐进切换为：

- 本地开发：仍可保留 `/uploads`
- 生产环境：对象存储作为主媒体存储
- 外部访问：CDN 分发

应用层只负责：

- 资产元数据
- 权限校验
- 签名下载
- 任务状态

#### B-2. 视频转码与切片

视频类资产统一进入异步媒体处理任务：

- 原始视频入库
- 生成 `poster`
- 生成预览版 mp4
- 生成 HLS 多码率切片
- 记录转码状态和可用清晰度

建议最终对外字段：

- `posterUrl`
- `previewVideoUrl`
- `hlsUrl`
- `downloadUrl`
- `availableQualities`

#### B-3. 自适应码率播放

前端播放优先消费 `hlsUrl`：

- 弱网自动切低码率
- 正常网络默认中码率
- 全屏或高带宽时自动切高码率

这样比“永远播放一个大 mp4”更符合真实用户体验。

#### B-4. 图片多尺寸与现代格式

图片类资产统一生成多尺寸和现代格式：

- 小图：卡片和列表
- 中图：详情预览
- 高清图：放大查看
- 原始图：下载

优先考虑：

- `WebP`
- `AVIF`
- 保留原始下载格式

#### B-5. 媒体任务中心

新增统一媒体处理任务类型：

- `image_derivative`
- `video_transcode`
- `video_poster_extract`
- `video_hls_package`
- `audio_waveform_extract`

所有媒体衍生任务统一由 Worker 承接，不挤占 Web 进程。

#### B-6. CDN 与缓存头治理

对外媒体访问统一走 CDN，策略分层：

- 公开预览资源：长缓存
- 受控下载资源：短时签名 URL
- 正在处理中资源：不缓存或短缓存

#### B-7. 前端交互升级

在方案 A 的基础上再升级：

- 视频支持清晰度切换
- 支持时间轴预览图
- 支持大图渐进放大
- 支持弱网策略和自动降级

### 5.4 适合当前项目的原因

这是当前项目的明确中期目标，原因如下：

- 项目本身就是 AIGC 影视化工作流，视频资产和媒体流量会持续增长。
- 后端已经在推进长任务 Worker 化、静态流量剥离和生产性能治理，这为媒体管线升级提供了基础。
- 一旦创作页、分镜页、资产库和剪辑成片形成高频回看，单靠本地 `/uploads` 很容易成为瓶颈。

### 5.5 方案 B 的主要收益

- 视频播放首帧更快
- 弱网播放更稳定
- 拖动与 seek 体验更好
- 应用层带宽与 IO 压力更低
- 媒体分发能力和扩容能力更强

### 5.6 方案 B 的代价

- 基础设施成本更高
- 后端任务编排更复杂
- 需要补充对象存储、CDN、转码、回源和签名下载方案
- 排障链路从“应用 + Nginx”升级为“应用 + Worker + 存储 + CDN + 转码任务”

## 6. 两套方案对比

| 维度 | 方案 A：轻量媒体分层交付 | 方案 B：生产级媒体资产管线 |
|---|---|---|
| 当前适配度 | 高 | 中 |
| 落地速度 | 快 | 中慢 |
| 改动风险 | 低 | 中高 |
| 对现有部署侵入 | 低 | 高 |
| 首屏/列表体验提升 | 高 | 高 |
| 视频长播放体验提升 | 中 | 高 |
| 弱网体验 | 中 | 高 |
| 大规模分发能力 | 低到中 | 高 |
| 运维复杂度 | 低 | 高 |

## 7. 推荐决策

### 7.1 当前正式推荐

当前正式推荐采用：

- 近期执行：方案 A
- 中期演进目标：方案 B+

也就是：

`先用方案 A 解决当前最痛的体验问题，再用升级后的方案 B+ 解决增长后的分发、播放、安全与治理上限问题。`

### 7.2 不推荐的路线

当前不推荐以下路线：

- 继续只做前端懒加载，但后端仍只给原始大文件
- 直接把所有媒体访问都切到完整对象存储 + HLS，而不先完成媒体字段语义治理
- 页面层继续自己猜 `url` 到底是缩略图、原图、预览图还是下载地址

## 8. 建议实施顺序

### 阶段 1：先落方案 A 的最小闭环

- 统一图片 `thumbnailUrl / previewUrl / downloadUrl`
- 统一视频 `posterUrl / previewVideoUrl / downloadUrl`
- 列表接口轻量化
- 详情接口重字段下沉
- 前端页面默认只消费轻预览

### 阶段 2：补后处理与状态承接

- 生成任务完成后追加派生产物任务
- 页面显示预览准备状态
- 下载入口与预览入口彻底分离

### 阶段 3：评估是否进入方案 B

满足以下任一条件时，进入方案 B 设计与落地：

- 视频资产量明显增长
- 用户反馈播放卡顿和拖动差仍然明显
- 外部分享访问和峰值回源压力增大
- 单机 `/uploads + Nginx` 已开始成为瓶颈

## 9. 后续设计约束

后续无论采用哪套方案，都建议继续遵守以下约束：

1. 页面层只消费适配层视图模型，不直接猜后端字段。
2. 预览、播放、下载必须分语义，不继续复用一个宽泛 `url`。
3. 列表、详情、弹窗、下载必须共享同一来源模型，不各自重猜。
4. 长任务返回的媒体结果，先承接任务状态，再承接派生产物状态。
5. 媒体体验优化优先做“交付分层”，不是先做视觉改版。

## 10. 一句话结论

对当前 miioo 项目来说，最合适的技术路线不是直接重做媒体基础设施，而是：

先以方案 A 的“轻量媒体分层交付”快速改善当前生成资产访问体验，把列表、详情、播放和下载职责拆清；待主链路与生产联调进一步稳定后，再按方案 B 升级到“对象存储 + CDN + 转码切片 + 自适应码率”的生产级媒体资产管线。
