# 分镜页面全能参考弹窗说明

## 1. 文档目的

这份文档用于基于当前仓库真实代码，梳理“项目模块分镜页面中的全能参考弹窗功能”到底是怎么工作的，重点回答 3 个问题：

1. 提示词是如何与数字资产挂钩的
2. 提示词是如何编排、如何展示给用户的
3. 前后端能力是如何打通的

本文档不讨论视觉改版，也不把当前代码中尚未落地的设想写成既成事实。

---

## 2. 真实入口在哪里

当前分镜页里的“全能参考”并不是一个单独命名的组件文件，而是分镜视频生成侧边面板中的一种模式。

真实入口位于：

- `frontend/src/pages/StoryboardPage.jsx`
- 核心函数：`GenerateVideoPanel(...)`

当前面板分为两个模式：

- `tab === 'all'`：全能参考
- `tab === 'frame'`：首尾帧

也就是说，用户在分镜页里点击“生成分镜视频”后，看到的侧边面板本身就是“全能参考弹窗”的真实承载层。

### 2.1 全能参考模式里有什么

在 `all` 模式下，页面会展示这些参考位：

- 参考主体
- 参考图
- 参考视频
- 参考音频

这些参考位都服务于同一个目标：把“文本提示词”从纯文本输入，升级成“带数字资产上下文的生成指令”。

### 2.2 首尾帧模式里有什么

在 `frame` 模式下，面板切换为：

- 首帧图
- 尾帧图

这说明当前分镜页已经把视频生成能力清晰拆成两种思路：

- 全能参考：更偏多模态条件控制
- 首尾帧：更偏镜头起止画面控制

这也是为什么文档里要把“全能参考”和“首尾帧”明确拆开讲，不能混为一个概念。

---

## 3. 这套功能的核心思路

分镜页里的“全能参考”本质上不是“多放几个上传按钮”，而是下面这套思路：

1. 先把镜头本身组织成基础提示词
2. 再给提示词补充可引用的数字资产上下文
3. 再根据当前模型支持的参考模式，把这些上下文转成后端真实可消费的参数
4. 最后由后端完成校验、生成、落盘和结果回写

从产品思路上看，它解决的是一个很实际的问题：

- 单纯文本提示词太抽象
- 用户往往已经有角色图、场景图、参考视频、参考音频
- 如果这些数字资产无法进入生成链路，提示词就只能“描述”，不能“约束”

所以“全能参考”的核心不是弹窗，而是让提示词从“描述型文本”升级成“文本 + 资产绑定”的组合输入。

---

## 4. 提示词如何与数字资产挂钩

当前仓库里，提示词与数字资产挂钩，实际上有两层实现。

## 4.1 分镜页当前的直接挂钩方式

分镜页在 `StoryboardPage.jsx` 里的做法偏直接，也最容易理解。

### 4.1.1 本地上传参考图

当用户在分镜图或分镜视频面板上传图片参考时，前端会：

1. 先上传文件
2. 拿到上传后的 URL
3. 把这个 URL 以标签形式追加进当前 prompt

当前采用的可见标记形式是：

```text
[参考图:实际URL]
```

这类标记有两个作用：

- 对用户可见，能让用户知道当前提示词已经带上了哪张参考图
- 对当前前端逻辑可消费，后续生成时可把这些参考图一并传给后端

### 4.1.2 从资产库选择参考图

当用户不是本地上传，而是从资产库选择参考图时，分镜页也采用同样思路：

1. 打开统一资产选择器
2. 选择项目资产
3. 回拿完整资产对象
4. 取出 `url / file_url / image url`
5. 同步更新参考图列表
6. 同步给 prompt 追加 `[参考图:URL]`

也就是说，分镜页当前的挂钩方式是“显式标签注入”。

它的优点是：

- 实现简单
- 用户可见
- 便于当前面板内即时回填

它的边界也很明显：

- 绑定语义主要靠 URL 标签表达
- 更偏当前面板局部状态
- 结构化程度不如创作页完整

## 4.2 当前仓库里更完整的结构化挂钩方式

如果只看分镜页，会觉得“提示词和资产挂钩”就是往 prompt 里拼 `[参考图:URL]`。

但仓库里其实已经有一套更完整的做法，落在创作页与统一生成链路中。

真实入口主要在：

- `frontend/src/pages/CreationPage.jsx`
- `frontend/src/api/creation.js`
- `backend/app/routers/creation.py`

创作页的 `InputCard` 会从富文本输入区提取 4 类核心数据：

- `promptText`
- `promptRaw`
- `mentions`
- `attachments`

其中：

- `promptText`：给用户看的纯文本语义
- `promptRaw`：保留 `@资产名` 这类原始输入痕迹
- `mentions`：记录提示词里具体提到了哪些资产、提及位置是什么
- `attachments`：记录这次参与生成的真实资产集合

这套设计比简单拼 URL 更强，原因是它把“提到了什么”和“真实绑定了什么”拆开了。

### 4.2.1 mentions 解决什么问题

`mentions` 解决的是“提示词里提到了谁”。

它记录的信息包括：

- `mention_id`
- `asset_id`
- `asset_type`
- `asset_name`
- `display_text`
- `start / end`
- `intended_role`

也就是说，后端并不是只收到一句模糊的 prompt，而是能知道：

- 用户在 prompt 的哪个位置提到了哪个资产
- 这个资产是图片、视频还是音频
- 这个提及本来打算承担什么参考角色

### 4.2.2 attachments 解决什么问题

`attachments` 解决的是“哪些数字资产真的参与了这次生成”。

它会统一承接：

- 资产库选中的资源
- 用户本地上传的资源
- 已经有 URL 的引用资源

并统一记录：

- `assetId`
- `assetType`
- `assetName`
- `url`
- `role`
- `source`

这样后端拿到的不是“几个零散字段”，而是一组可校验、可回写、可持久化的资产绑定数据。

### 4.2.3 prompt_resolved 为什么重要

前端和后端都会参与构建 `prompt_resolved`。

它不是简单复读用户输入，而是“把原始提示词和资产上下文融合后的最终提示词”。

当前逻辑会把提及的资产整理成类似这样的补充说明：

```text
重点参考资产：
1. 角色A（image），请优先参考其主体、动作、构图或风格特征
2. 背景音乐（audio），请优先参考其内容与语义特征
```

这样做的意义是：

- 对上游模型更明确
- 对后端回写更稳定
- 对刷新后结果复盘也更友好

## 4.3 对分镜页的启发

因此，分镜页当前虽然主要还是 URL 级挂钩，但其设计思路已经非常明确：

- UI 上先用最直接的方式把参考资产挂进提示词和表单状态
- 能力思想上对齐创作页那套更成熟的“mentions + attachments + prompt_resolved”结构化方案

换句话说，分镜页现在已经有“入口形态”，创作页和后端统一链路则提供了“完整能力范式”。

---

## 5. 提示词是如何编排的

提示词编排不是只有一个输入框那么简单，它至少分成 3 层。

## 5.1 第一层：镜头基础信息自动拼装

分镜页里，初始 prompt 并不是凭空开始写的，而是从当前 shot 自动生成基础内容。

当前事实源包括：

- `buildPromptFromShot(...)`
- `backend/app/routers/storyboards.py` 中的 `_build_storyboard_composed_prompt(...)`

后端已经具备较完整的结构化拼装能力，会把当前镜头信息拆成若干段：

- 镜头内容
- 主体与场景
- 镜头语言
- 光影氛围
- 声音信息
- 风格与一致性

这意味着当前系统中的“分镜提示词”并不是随意输入的一段话，而是默认由镜头结构化信息长出来的。

## 5.2 第二层：人工补充与修订

用户打开分镜视频生成面板后，可以继续修改当前 prompt。

这时的 prompt 已经不只是后端拼好的自动稿，还会叠加：

- 用户自己补充的画面描述
- 额外强调的动作、氛围、风格
- 参考图 URL 标签
- 视频/音频等额外参考语义

也就是说，真正送给模型的提示词，通常是“自动编排 + 用户修订”的混合稿。

## 5.3 第三层：前后端共同收口的最终稿

在分镜和创作统一链路里，最终会收口出几个不同层次的 prompt：

- `prompt`：当前直接提交文本
- `prompt_raw`：保留原始输入痕迹的版本
- `prompt_resolved`：融合资产上下文后的最终版本

前端适配层 `frontend/src/api/storyboard.js` 还会把后端沉淀的提示词状态暴露为：

- `composedPromptAuto`
- `composedPromptManual`
- `composedPromptDirty`
- `composedPromptSourceVersion`

这几个字段说明当前系统已经不是“一份 prompt 走天下”，而是显式区分：

- 自动生成稿
- 人工修订稿
- 人工稿是否已偏离自动稿
- 当前自动稿对应的源版本

## 5.4 这套编排对展示有什么影响

这套编排方式直接影响了页面展示效果。

因为用户在面板里看到的不再只是一个普通输入框，而是一个承接了 3 类信息的控制区：

1. 镜头原始语义
2. 用户人工补充
3. 数字资产上下文

这会带来两个结果：

- 用户更容易理解“我为什么会得到这个生成结果”
- 页面在刷新回显、历史结果复盘、二次编辑时也更容易拿到稳定事实

---

## 6. 提示词的展示效果为什么是现在这样

从页面设计角度，当前“全能参考”面板把这些内容放在同一处，其实是在强化一种产品认知：

- 提示词不是孤立文本
- 参考图、参考视频、参考音频不是附属附件
- 模型选择、参考模式、提示词、结果区应被看作同一条生成链路

## 6.1 为什么参考位要显式展示

如果只把文件悄悄上传给后端，而不在面板里显式展示，用户会有两个问题：

- 不知道当前到底带了哪些参考素材
- 不知道这次生成失败或偏题时该排查哪个条件

当前面板显式展示：

- 参考主体
- 参考图
- 参考视频
- 参考音频
- 首尾帧

本质上是在把“隐藏上下文”变成“可见上下文”。

## 6.2 为什么结果区要和表单区并排

分镜页当前把结果区放在同一个侧边面板右侧，也是有明确目的的：

- 让用户边看结果边改提示词
- 让选中的历史结果回填左侧参数
- 让“提示词 -> 结果 -> 再编辑”形成闭环

这和主体页历史图区的设计方向是一致的，强调的是“同一面板内复盘与迭代”。

## 6.3 为什么这种设计适合分镜场景

分镜视频不是一次性创作，而是高度迭代型工作。

用户通常会反复调整：

- 当前镜头提示词
- 参考主体
- 参考图
- 参考视频
- 参考音频
- 时长
- 分辨率
- 模型

所以当前的展示方式，本质上是在服务“快迭代”而不是“只提交一次”。

## 7. 前端是如何把能力打到后端的

全能参考的前后端打通，不是一个接口完成的，而是一个分层链路。

## 7.1 前端面板层负责收集输入

分镜页 `GenerateVideoPanel(...)` 负责采集当前生成所需的参数：

- `prompt`
- `resolution`
- `duration`
- `reference_images`
- `reference_video_url`
- `reference_audio_url`
- `first_frame_url`
- `last_frame_url`
- `reference_mode`

其中：

- 全能参考模式下，重点是 `reference_images / reference_video_url / reference_audio_url`
- 首尾帧模式下，重点是 `first_frame_url / last_frame_url`

## 7.2 前端适配层负责翻译 UI 语义

前端不会把 UI 文案直接原样扔给后端，而是通过适配层做一次翻译。

真实翻译层主要在：

- `frontend/src/utils/modelAdapter.js`
- `frontend/src/api/storyboard.js`
- `frontend/src/api/creation.js`

这里解决的关键问题有两个。

### 7.2.1 UI 的“全能参考”不是后端真实枚举

前端面板里展示给用户的是：

- 全能参考
- 首尾帧

但后端真实消费的往往是更底层的 `reference_mode`，例如：

- `full`
- `video_ref`
- `first_frame`

因此 `modelAdapter.js` 会根据模型能力把前端的抽象选项映射到后端真实值。

这一步非常重要，因为不同模型支持的参考模式并不完全一样。

### 7.2.2 资产字段需要统一归一

前端资产来源并不只有一种：

- 本地上传文件
- 资产库已存在资源
- 已有 URL 的引用资源

适配层会把这些来源统一归一为后端可稳定消费的字段组合，避免页面层自己到处写兼容逻辑。

## 7.3 后端路由负责校验与承接

后端主要由两个路由层承接：

- `backend/app/routers/creation.py`
- `backend/app/routers/storyboards.py`

后端会做这些事情：

1. 解析 `mentions`
2. 解析 `attachments`
3. 校验当前模型支持哪些参考模式
4. 校验当前模型支持哪些资产绑定
5. 推导真实 `reference_mode`
6. 处理首尾帧、参考视频、参考音频
7. 生成最终 prompt
8. 调用视频生成服务
9. 落盘外部媒体
10. 把结果写回资产与任务元数据

也就是说，后端不是“被动存一下参数”，而是在真正完成“结构化承接 -> 模型校验 -> 结果持久化”的闭环。

## 7.4 后端为什么要沉淀 metadata

后端在生成完成后，会把很多关键上下文写回 `metadata_json` 或任务参数中，例如：

- `reference_mode`
- `prompt_raw`
- `prompt_resolved`
- `asset_bindings`
- `first_frame_url`
- `last_frame_url`
- 缩略图和预览相关字段

这样做的价值非常高，因为它让后续这些能力成为可能：

- 刷新页面后恢复历史事实
- 打开详情时回显本次生成上下文
- 重新编辑时拿到真实上一轮参数
- 排查问题时能知道这次到底带了哪些参考资源

---

## 8. 数字资产在这条链路里分别扮演什么角色

当前分镜页中的数字资产，并不是“统称附件”，而是承担不同生成角色。

## 8.1 图片类资产

图片类资产通常承担：

- 角色参考
- 场景视觉参考
- 构图参考
- 风格参考
- 首帧或尾帧控制

## 8.2 视频类资产

视频类资产通常承担：

- 动作参考
- 镜头运动参考
- 节奏或演出参考

## 8.3 音频类资产

音频类资产通常承担：

- 语气
- 节奏
- 音色
- 氛围参考

这也是为什么“全能参考”比“单纯参考图”更强：它让系统不仅能参考画面，还能参考动作和声音语义。

---

## 9. 当前实现的边界

虽然当前链路已经很完整，但仍要明确边界，避免误读。

## 9.1 分镜页本身仍以面板局部状态为主

分镜页当前对资产绑定的承接，更多还是：

- 面板本地 state
- URL 级参考图标签
- 当前提交请求的字段组装

它还没有像创作页那样，在 UI 输入层完整展开为富文本 mention 系统。

## 9.2 完整结构化方案主要体现在创作链路

真正更成熟的“提示词和数字资产绑定”结构，目前仍主要沉在：

- 创作页输入区
- `api/creation.js`
- `creation.py`

所以如果后续要继续增强分镜页，全局最值得复用的不是再新增更多临时字段，而是直接吸收这套结构化绑定协议。

## 9.3 当前文档的结论不能扩大解读

本文档说明的是“当前仓库已经真实存在的思路与链路”，不代表：

- 分镜页已经完整复用了创作页的全部 mention 输入能力
- 所有视频模型都支持同等级的全能参考
- 所有资产类型都能在任意模型下自由混用

真正是否支持，还要以当前模型能力表和后端校验结果为准。

---

## 10. 一句话总结

分镜页面里的“全能参考弹窗”本质上是一套多模态生成控制面板。

它的关键不在“弹窗”本身，而在于：

- 用分镜结构化信息生成基础提示词
- 用数字资产把提示词从纯文本升级成可约束的多模态指令
- 用前端适配层把 UI 语义翻译成后端真实参数
- 用后端统一完成校验、生成、落盘与元数据回写

因此，这个功能真正沉淀出来的不是一个局部页面技巧，而是一条完整的“提示词 + 数字资产 + 模型能力 + 任务结果”的生成链路。
