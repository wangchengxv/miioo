# 生成资产前端访问体验方案 B+：渐进增强的企业级媒体资产管线升级

## 1. 文档定位

本文档用于把当前 `miioo` 项目在“生成资产访问体验”方向上的中期升级路线，从原先偏概览式的方案 B，升级为更适合本项目现实的 `B+ 渐进增强` 企业级方案。

配套执行拆分文档见：

- `docs/plans/生成资产前端访问体验方案B+实施清单.md`

它负责：

- 基于当前仓库真实代码与文档事实，定义后续媒体资产平台的目标边界
- 在方案 A 已建立的媒体字段语义与轻量分层基础上，补齐企业级生产方案的安全、稳定、体验、治理设计
- 为后续后端、前端、运维、Runbook、文档回写提供统一事实源

它不负责：

- 直接替代对象存储、CDN、转码集群的部署手册
- 直接替代未来具体接口实现文档
- 在当前阶段强行推动“一步到位全量重构”

一句话讲，`B+` 不是“推翻方案 A 再另起一套重平台”，而是把当前仓库已经建立的媒体语义、Worker 化方向、`Nginx` 静态承载基线和 Harness 文档治理，一起升级为可渐进落地的企业级媒体资产管线方案。

## 2. 当前项目事实基线

基于当前仓库，和本方案直接相关的事实如下：

- 当前生产媒体主链路仍以本地 `/uploads` 为主，但生产基线已明确倾向由 `Nginx` 而非 FastAPI 应用层承载热点静态流量。
- 方案 A 已在后端与前端建立了图片 `thumbnailUrl / previewUrl / downloadUrl`、视频 `posterUrl / previewVideoUrl / downloadUrl` 的媒体分层语义。
- 后端 `backend/app/services/media_derivative_pipeline.py` 与 `backend/app/services/media_view_models.py` 已具备轻量派生产物和媒体字段统一收口能力。
- 前端展示约束已明确要求页面只消费 `frontend/src/api/` 返回的视图模型，列表、详情、预览、下载必须共享同一来源模型。
- 生产环境并发与性能治理计划已将共享状态、长任务、轮询、媒体流量、观测等问题收口为专项 Harness 计划。
- 当前线上已具备 `miioo-web / miioo-worker` 的 Supervisor 托管基线，这意味着后续“Web 与异步媒体处理拆层”有现实承接基础。

因此，本项目现在缺的并不是“是否要上媒体平台”这一层认知，而是缺一份贴合现状的企业级升级方案，把“什么时候上”“先上什么”“如何避免过度设计”“如何用 Harness 承接全过程”讲清楚。

## 3. 为什么方案 B 要升级为 B+

原方案 B 已经指出对象存储、CDN、转码和 HLS 是中期方向，但若直接按“基础设施名词清单”推进，仍存在四个不足：

- 安全边界不够细：还没有明确区分公开预览、登录态预览、受控下载、内部回源、管理回查等不同访问域。
- 稳定性设计不够细：还没有把任务幂等、失败补偿、断点恢复、派生产物一致性和回滚策略写成明确规则。
- 治理设计不够细：还没有把容量预算、SLO、指标、Runbook、灰度、文档回写矩阵收口成可执行工程约束。
- 与当前项目连接不够紧：还没有明确说明如何承接方案 A 已完成的字段语义、页面适配、Worker 方向和 Harness 文档体系。

因此，本次升级后的 `B+` 结论不是“比方案 B 更重”，而是“比原方案 B 更安全、更稳定、更可渐进落地，也更符合本项目的 Harness 工程方法”。

## 4. B+ 总体目标

把当前“本地文件 + 预览分层”的媒体交付方式，升级为“具备存储分层、访问分层、转码分层、权限分层、治理分层”的企业级媒体资产管线。

最终希望达到的状态是：

- 应用层只负责业务元数据、权限校验、任务状态和签名发放
- 媒体实际分发由对象存储 + CDN 承担
- 媒体处理由独立 Worker 与队列承担，而不是继续挤占 Web 进程
- 前端按统一视图模型消费 `thumbnail / preview / hls / download` 等显式字段
- 媒体体验、访问安全、生产稳定性和 Harness 文档治理形成一个完整闭环

## 5. B+ 设计原则

### 5.1 渐进增强，不一步到位全量替换

- 本地开发仍允许 `/uploads`
- 已落地的方案 A 媒体语义继续保留
- 前端视觉与页面交互骨架不因本方案而被重做
- 新基础设施按阶段引入，不把当前联调主线扩散成“大迁移项目”

### 5.2 安全、稳定、体验、治理四条线并行

- 不允许只上对象存储和 CDN，却不补权限、签名、审计和生命周期治理
- 不允许只上播放器体验，却不补任务状态机、失败补偿和观测
- 不允许只谈架构图，不回写 Runbook、进度文档和变更日志

### 5.3 前端继续只消费视图模型

- 页面层不直接猜后端字段
- 页面层不自己拼接媒体访问策略
- 页面层只消费适配层定义好的预览、播放、下载、清晰度和状态字段

### 5.4 应用层职责继续收口

应用层负责：

- 资产元数据
- 权限与签名
- 任务状态
- 派生产物登记
- 生命周期与审计

应用层不再继续直接承担：

- 热点静态媒体大流量
- 播放器 seek 的高频 Range 流量
- 大体积视频长时间直出

## 6. 企业级运行架构

建议目标架构拆为以下职责层：

- `Web/API 层`：创建资产、查询资产、校验权限、发放签名、返回视图模型
- `Queue / Worker 层`：图片派生、视频转码、海报提取、HLS 切片、波形提取、清理任务
- `Object Storage`：原始文件、预览文件、派生产物、切片包、私有下载文件
- `CDN`：公开预览分发、播放器回源、边缘缓存、Range 支持、缓存头治理
- `Metadata DB`：资产记录、派生产物状态、任务状态、质量档位、审计信息
- `Redis / Cache`：共享状态、队列协调、短时签名辅助态、热点元数据缓存
- `Observability`：任务时延、队列堆积、回源错误、播放失败、CDN 命中率、下载审计

当前项目与目标态之间的桥接方式应为：

- 本地开发：保留 `/uploads`
- 生产第一阶段：`Nginx + /uploads` 承担静态直出，应用层只做元数据
- 生产第二阶段：原始文件和派生产物逐步切入对象存储
- 生产第三阶段：对外播放和预览统一经 CDN，应用层只下发受控地址与元数据

## 7. 安全与权限分层

### 7.1 资源访问等级

建议将媒体资源至少拆成四类：

- `公开预览资源`
  - 如公开海报、小图、低风险短时预览
  - 可走长缓存 CDN
- `登录态预览资源`
  - 仅登录用户可访问的预览图、预览视频、波形图
  - 适合短时签名或鉴权网关校验
- `受控下载资源`
  - 原图、原视频、原音频、导出包
  - 必须短时签名，并记录下载审计
- `内部回源资源`
  - 转码中间产物、回查文件、任务临时目录
  - 不直接暴露给前端

### 7.2 签名与权限策略

- `downloadUrl` 不应长期为固定裸地址
- 原始下载建议默认改为短时签名 URL
- `previewUrl / previewVideoUrl / hlsUrl` 按资源敏感度决定是否签名
- 签名需绑定：
  - 用户身份
  - 项目作用域
  - 资源 ID
  - 过期时间
  - 可选的单次下载限制

### 7.3 安全控制点

- 防盗链：校验签名、来源域、过期时间和资源作用域
- 防越权：同一用户不能通过猜测路径访问其它项目媒体
- 防误分享：签名默认短效，不把受控下载长期暴露成稳定外链
- 可审计：记录原始下载、批量导出、异常频繁访问、签名发放来源
- 可删除：当用户删除资产或触发合规删除时，原始文件、派生产物、缓存与 CDN 失效链路要可追踪

## 8. 存储层升级

### 8.1 分桶建议

对象存储建议按职责分层，而不是全部堆进一个目录：

- `raw`
  - 原始上传文件、模型返回原始结果
- `preview`
  - 中清预览图、预览 mp4、音频预听版本
- `derived`
  - 缩略图、海报图、时间轴预览图、波形图
- `hls`
  - `m3u8 + ts/fmp4` 切片产物
- `private-download`
  - 受控下载、导出 zip、临时分发文件

### 8.2 生命周期策略

- `raw`：长保留，按业务保留期限与成本策略处理
- `preview / derived`：可重建，适合按冷热分层和 TTL 管理
- `hls`：高热点时长缓存，冷资源可回收重建
- `private-download`：短生命周期，签名过期后可清理

### 8.3 删除与合规策略

- 删除资产时，不只删除元数据
- 需要同步明确：
  - 原始文件删除策略
  - 派生产物清理策略
  - CDN 失效策略
  - 审计留痕策略

## 9. 媒体处理运行时与一致性设计

### 9.1 统一媒体任务中心

建议统一媒体处理任务类型：

- `image_derivative`
- `video_poster_extract`
- `video_preview_transcode`
- `video_hls_package`
- `audio_waveform_extract`
- `media_cleanup`

所有衍生任务统一由独立 Worker 承接，不再继续挤占 Web 进程。

### 9.2 任务状态机

建议媒体派生状态至少包含：

- `queued`
- `running`
- `partial_ready`
- `ready`
- `failed`
- `retrying`
- `cancelled`

其中：

- `partial_ready` 表示已有可用预览，但更高质量产物仍在处理中
- `ready` 表示当前阶段承诺的派生产物全部可用

### 9.3 幂等与补偿

- 同一原始文件进入派生链路时，应存在幂等键
- 重复提交不得生成多份相同派生任务
- 转码失败需支持：
  - 自动重试
  - 人工重驱动
  - 死信留存
  - 失败原因可追踪

### 9.4 一致性要求

禁止出现以下状态：

- 数据库标记 `ready`，但实际预览文件或 HLS 清单不可访问
- 卡片可见 `posterUrl`，但详情播放地址缺失
- 下载入口仍指向预览文件或失效旧文件

因此建议把“文件写入成功”“对象存储可读”“CDN 可回源”“数据库状态更新”作为连续步骤串起来，至少保证：

- 先确认产物可读
- 再回写资产元数据
- 最后切换 `previewReady / derivativeStatus / availableQualities`

## 10. 视频转码与播放分层

### 10.1 视频产物层级

视频类资产建议至少拆成：

- `posterUrl`
  - 列表、详情初始静态占位
- `previewVideoUrl`
  - 详情默认播放的轻量 mp4
- `hlsUrl`
  - 中长期主播放地址
- `downloadUrl`
  - 原始下载地址
- `availableQualities`
  - 当前可用清晰度集合

### 10.2 HLS 推进策略

不要求一开始就上最重的多码率方案，可按阶段推进：

- 第一阶段：单码率 HLS，先解决播放器 seek 和分片加载问题
- 第二阶段：多码率 ABR，补 `360p / 540p / 720p / 1080p`
- 第三阶段：时间轴预览图、播放器清晰度切换、弱网自适应

### 10.3 转码规则建议

- 优先生成可快速首帧展示的 `poster`
- 优先生成轻量 `previewVideoUrl`，避免详情直接拉原始大文件
- HLS 切片队列优先级高于“非关键体验衍生物”
- 转码失败时，前端可降级回 `previewVideoUrl`，但不能直接把原始大文件当默认主播放地址

## 11. 图片多尺寸与现代格式

图片类资产建议统一生成：

- `thumbnail`
  - 列表卡片、宫格、轻量预览
- `preview`
  - 详情弹窗、中屏预览、编辑回显
- `large`
  - 放大查看、细节预览
- `original`
  - 原始下载

建议优先支持：

- `WebP`
- `AVIF`
- 原始格式保留下载

建议同时补齐：

- 尺寸命名规范
- 格式回退策略
- 浏览器兼容协商策略
- 大图渐进加载策略

## 12. CDN 与缓存头治理

### 12.1 缓存策略分层

- 公开预览资源：长缓存 + 版本化路径
- 登录态预览资源：短缓存 + 受控签名
- 受控下载资源：极短缓存或不缓存
- 转码中资源：不缓存或非常短缓存

### 12.2 回源治理

- 预热高频资源
- 保护源站，避免播放峰值直接打到应用层
- 支持播放器 Range 请求和 seek 场景
- 对签名失效、回源失败、CDN 404 设计明确的回退和告警路径

### 12.3 缓存键策略

缓存键至少应考虑：

- 资源版本
- 质量档位
- 图片尺寸
- 授权级别
- 是否签名资源

## 13. 前端消费与体验规范

在方案 A 的基础上，前端升级重点不是改 UI，而是升级媒体消费策略：

- 卡片仍优先展示 `thumbnailUrl / posterUrl`
- 详情优先展示 `previewUrl / previewVideoUrl`
- 播放优先消费 `hlsUrl`，失败再回退到 `previewVideoUrl`
- 下载统一走 `downloadUrl`
- 清晰度切换只依赖显式的 `availableQualities`
- 大图查看走渐进加载，不直接默认拉原始图
- 弱网下默认更保守的预览策略

前端继续遵守：

- 页面层只消费 `src/api/` 视图模型
- 页面层不直接猜媒体类型
- 列表、详情、弹窗、下载必须共享同一来源模型

## 15. 与方案 A 的关系

`B+` 不是替代方案 A，而是建立在方案 A 基础上的企业级升级路线。

先做方案 A 的原因仍然成立：

- 先把媒体字段语义理顺
- 先把列表、详情、下载职责拆开
- 先让当前用户感知体验明显改善

而进入 `B+` 的前提，是在方案 A 的语义基线上继续补齐：

- 存储分层
- 权限分层
- 派生任务状态机
- CDN 分发
- 观测与 Runbook

这样后续升级时，才不会一边上基础设施，一边还在补最基础的媒体字段定义。

## 16. 适用阶段

以下情况出现时，适合正式从方案 A 进入 `B+`：

- 当前主链路和媒体语义已经稳定
- 真实环境用户量、媒体量和播放量明显增长
- 用户反馈播放卡顿、拖动差、首帧慢的问题仍然明显
- 外部分享访问和峰值回源压力增大
- 单机 `/uploads + Nginx` 已开始成为瓶颈
- 需要更高质量的权限治理、签名下载、审计和容灾能力

## 17. 推荐实施顺序

### 阶段 0：语义与契约基线冻结

- 冻结图片/视频/音频对外媒体字段语义
- 冻结预览、播放、下载的职责分层
- 冻结前端适配层消费规则
- 盘点现有 `/uploads`、派生产物、下载口径和历史脏数据

### 阶段 1：存储与访问分层

- 明确对象存储桶分层
- 明确 CDN 域名与回源策略
- 明确预览资源与下载资源的权限边界
- 明确签名 URL 生命周期与失效规则

### 阶段 2：媒体处理运行时与状态机

- 视频转码任务化
- 海报图提取
- 预览版 mp4 生成
- HLS 切片生成
- 波形图、时间轴预览图等衍生物规则化
- 任务幂等、重试、死信与人工重驱动设计

### 阶段 3：播放与分发体验升级

- 前端播放器支持 HLS
- 支持清晰度切换
- 支持时间轴预览图
- 支持弱网自动降级
- 支持大图渐进放大和现代格式回退

### 阶段 4：观测、成本、容灾与治理闭环

- 建立媒体链路 SLO 与容量预算
- 补齐任务队列长度、转码成功率、CDN 命中率、回源错误率指标
- 完成签名、回源、转码与下载链路 Runbook
- 完成灰度、回滚、故障切换与成本复盘机制

## 18. 方案收益

`B+` 相比原始方案 B，收益不只在播放体验，还在工程稳定性：

- 视频首帧更快
- 弱网播放更稳定
- 拖动和 seek 体验更好
- 应用层带宽与 IO 压力更低
- 媒体分发能力更强
- 下载链路更安全
- 任务链路更可观测
- 故障排查路径更清晰
- 后续扩容与降本路径更可预测

## 19. 方案代价

`B+` 的主要代价：

- 基础设施成本更高
- 运行时链路更复杂
- 需要补对象存储、CDN、转码、签名、回源、观测和 Runbook
- 排障链路从“应用 + Nginx”升级为“应用 + Worker + 存储 + CDN + 队列 + 转码任务”

因此本方案的关键不在于“是否复杂”，而在于“是否按阶段推进、是否用 Harness 把复杂度治理住”。

## 20. 验收标准与上线门槛

在进入正式落地前，建议至少按五类标准验收：

### 20.1 安全

- 原始下载不再长期裸露直链
- 项目级、用户级资源访问边界可校验
- 下载具备审计留痕

### 20.2 稳定性

- 关键派生任务具备幂等、重试和失败补偿
- 不再出现“状态 ready 但产物不可读”的一致性断点
- 转码与分发失败可快速定位

### 20.3 体验

- 详情默认不再直接拉原始大文件
- 视频支持更快首帧与更稳定播放
- 大图与长视频在弱网下具备合理降级

### 20.4 运维

- 有明确容量预算
- 有明确指标面板
- 有明确故障排查 Runbook

### 20.5 Harness 回写

- 本方案推进过程中的关键结论已同步回写进度文档
- 相关运行手册、变更日志和必要接口文档已同步更新

## 21. 当前结论

对当前 `miioo` 项目来说，更安全、稳定又高效的生产企业级方案，不是跳过方案 A 直接做一个“重型媒体平台”，而是把原方案 B 升级为 `B+ 渐进增强`：

- 继续建立在当前媒体字段语义、前端适配规则和 Worker 化方向之上
- 在对象存储、CDN、转码、HLS 之外，补齐权限、签名、任务状态机、一致性、观测、Runbook 与 Harness 文档治理
- 让后续媒体基础设施升级不再是孤立的技术选型，而是一条可分阶段执行、可交接、可验收、可回滚的工程路线

因此，`B+` 应作为当前项目在生成资产访问体验方向上的中期正式目标方案写入设计边界，并作为后续相关实现和运行治理的统一事实源。
