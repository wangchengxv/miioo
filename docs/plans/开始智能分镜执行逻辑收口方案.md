# 开始智能分镜执行逻辑收口方案

## 1. 文档定位

本文档用于收口当前仓库中“开始智能分镜”按钮之后的真实执行链路。

- 参考来源：`/Users/xingyi/Desktop/迭代一版/storyboard-extraction-logic.md`
- 当前事实源：
  - `backend/app/routers/storyboards.py`
  - `backend/app/services/storyboard_gen.py`
  - `frontend/src/api/storyboard.js`
  - `frontend/src/pages/Home.jsx`
  - `frontend/src/pages/SubjectPage.jsx`
  - `frontend/src/pages/StoryboardPage.jsx`

说明：

- `storyboard-extraction-logic.md` 记录的是旧版“单集直返”思路，可作为抽镜提示词、story beat、目标镜头数和 JSON 修复链路的参考。
- 当前生效口径以本仓库现有代码为准，主入口已经收口为“基于定稿主剧本的批量后台任务”，不是旧文档里的主体页单集同步生成。

## 2. 当前统一语义

“开始智能分镜”当前统一表示：

- 基于项目当前已定稿的主剧本
- 必要时先重新拆分正式分集
- 再按正式分集批量生成分镜
- 旧分镜允许被覆盖式重跑
- 整体以后台长任务方式执行，不把任务创建响应误当最终结果

因此页面侧的统一承接方式必须是：

1. 启动批量任务
2. 拿到 `taskId`
3. 轮询任务状态
4. 任务进度中持续刷新分集/分镜结果
5. 终态后按真实接口结果回显

## 3. 真实执行链路

### 3.1 前端入口

- `frontend/src/pages/SubjectPage.jsx`
  - 工具栏按钮负责触发 `onStartStoryboard`
  - 已有分镜时先弹确认框，明确“覆盖全部正式分集旧分镜”
  - 若已有进行中的分镜任务，则不重复创建，只继续跳转分镜页查看进度

- `frontend/src/pages/Home.jsx`
  - `onStartStoryboard` 中先切到 `storyboard` 工作流步骤
  - 再执行 `handleGenerateStoryboards()`
  - `handleGenerateStoryboards()` 会优先：
    - 复用当前内存中的进行中任务
    - 查找当前项目最近的未终态批量分镜任务并恢复
    - 确认都没有后才新建任务

### 3.2 前端 API 适配层

- `frontend/src/api/storyboard.js`
  - `apiStartStoryboardGenerationFromFinalScript(projectId)` 负责启动批量任务
  - `pollStoryboardGenerationTask(projectId, taskId)` 负责轮询
  - `apiGetLatestStoryboardGenerationTask(projectId)` 负责恢复最近未终态任务
  - `normalizeStoryboardGenerationTask(task)` 负责把任务字段归一为页面可直接消费的视图模型

当前页面统一消费的关键任务字段包括：

- `stageLabel`
- `statusMessage`
- `currentEpisodeId`
- `currentEpisodeNumber`
- `currentEpisodeTitle`
- `lastCompletedEpisodeId`
- `lastCompletedEpisodeNumber`
- `lastCompletedEpisodeTitle`
- `completedEpisodeNumbers`
- `failedEpisodeNumbers`
- `warningMessages`
- `overwriteExisting`
- `results[].storyboardCount`
- `results[].error`

### 3.3 后端任务入口

- `backend/app/routers/storyboards.py`
  - `POST /api/projects/{project_id}/storyboards/generate-from-final-script`

该接口当前行为：

- 先检查主剧本是否存在
- 立即创建 `GenTask(task_type="storyboard_generate")`
- 立即返回任务对象
- 再由后台继续执行 `_continue_generate_storyboards_from_final_script(...)`

### 3.4 后端后台执行顺序

当前后台任务阶段收口为：

1. `queued`
2. `validating_final_script`
3. `finalizing_script`
4. `loading_subject_context`
5. `generating_episode_{n}`
6. `finalizing_results`
7. `completed / partial / failed`

每个阶段都会更新：

- `current_stage`
- `stage_label`
- `status_message`
- `queued_episode_numbers`
- `completed_episode_numbers`
- `failed_episode_numbers`
- `current_episode_*`
- `last_completed_episode_*`
- `warning_messages`
- `total_storyboard_count`

## 4. 抽镜质量如何承接旧文档思路

虽然当前入口已经从旧文档的“单集同步生成”演进为“整稿批量任务”，但抽镜能力仍然延续了旧文档里有价值的思路：

- `backend/app/services/storyboard_gen.py`
  - `_extract_story_beats()`
  - `_estimate_target_shot_count()`
  - `_build_user_prompt()`
  - `_call_storyboard_llm()`
  - `_post_process_shots()`

当前已将以下抽镜元信息沉到 `Storyboard.gen_params`：

- `beat_refs`
- `story_beat_count`
- `target_shot_count`
- `episode_number`
- `episode_title`
- `generation_source`

这样既保留了旧文档中的抽镜质量设计，又避免页面层继续靠聊天记忆或服务内隐式逻辑猜这次分镜为什么这么生成。

## 5. 页面侧最新收口点

### 5.1 `Home.jsx`

当前已升级为任务会话控制器：

- 负责统一启动任务
- 负责恢复当前项目最近未终态任务
- 负责避免同一任务重复轮询
- 轮询期间持续强刷 `episodes`
- 终态时统一给出 `completed / partial / failed` 摘要

### 5.2 `SubjectPage.jsx`

当前已升级为真实长任务入口：

- 按钮支持 `loading / disabled`
- 已有任务时不重复提交
- 已有分镜时先弹覆盖确认
- 文案已明确“后台长任务 + 可在分镜页查看逐集进度”

### 5.3 `StoryboardPage.jsx`

当前已从“全屏阻塞式等待”继续收口为“边生成边可看”：

- 若当前没有任何可展示分镜，继续使用整页 loading
- 若任务已有部分成功分集，则不再整页阻塞
- 页面顶部展示轻量任务提示条
- 提示条会显示：
  - 阶段文案
  - 已处理集数 / 总集数
  - 当前正在生成的分集
  - 最近完成的分集
  - 是否为覆盖式重跑
  - warning
  - 失败分集与错误原因
- 分集选择器已补齐任务状态标记：
  - `生成中`
  - `已完成`
  - `失败`
- 当当前分集没有分镜、但任务已有成功分集时，页面会自动切到首个可看的成功分集
- 若用户已手动切换分集，则不再强制覆盖用户选择

## 6. 当前任务状态承接规则

为避免后续页面继续各自实现一套状态机，当前统一采用以下口径：

- `completed`
  - 全部正式分集生成成功
  - 页面显示完成提示
- `partial`
  - 有成功分集，也有失败分集
  - 页面保留成功分镜可继续查看/编辑
  - 页面展示失败集清单，不走整页失败态
- `failed`
  - 当前批量任务没有产生可展示结果
  - 页面可展示全屏失败态和重试入口

## 7. 已知边界

- 本轮不重做分镜页整体视觉，只做状态承接与信息补齐
- 本轮不新增数据库迁移，继续复用 `GenTask` 和 `Storyboard.gen_params`
- 本轮不扩新的“分镜历史图定稿持久化”接口
- 本轮默认由用户手工做真实环境联调
