## 2. 项目背景与现状

`miioo` 当前不是一个从零启动的新项目，而是一个已经形成前端产品界面、后端基础能力、专项实施文档与阶段性联调成果的在研系统。

当前项目的真实背景如下：

- 前端页面和交互先行，产品工作流已经基本成型
- 后端接口在联调过程中逐步补齐，已具备较完整业务骨架
- 仓库已经形成 `CLAUDE.md + AGENTS.md + HARNESS_* + docs/` 的治理体系
- 核心主链路已经从“只有 mock 和占位”推进到“代码主干已接通、进入真实联调与专项收口”

当前系统重点目标不是推倒重做，而是：

- 保持现有 UI 和页面语义
- 在 `frontend/src/api/` 收口前后端契约翻译
- 补齐长任务运行时、媒体链路、模型能力事实源
- 形成适合长期维护和交接的工程文档体系

## 3. 业务可行性

### 3.1 用户价值可行

该项目服务的核心用户是影视创作者，目标是提供一条完整的 AIGC 影视化工作流：

- 从创意到剧本
- 从剧本到主体
- 从主体到分镜
- 从分镜到图片、视频、配音素材
- 从素材到剪辑成片

这一链路具备明显业务可行性，原因在于：

- 它覆盖的是完整创作流程，而非单点工具
- 上游模块输出可直接成为下游模块输入，链路天然有复用价值
- 用户对“减少跨工具切换、减少手工整理素材、减少结构化拆解成本”有明确需求

### 3.2 产品方向可行

当前项目的产品方向并未被后续联调推翻，反而已经被仓库现状反复证明：

- 首页、项目、新建项目、全局设定、剧本、主体、分镜、创作、资产、剪辑都已有明确页面事实
- 大多数专项计划都围绕“补齐真实闭环”而非“重做产品形态”
- 当前主要矛盾已经从“有没有页面”转向“如何让真实能力稳定承接”

这说明产品方向本身是稳定且可继续实施的。

### 3.3 交付路径可行

根据现有 `HARNESS_P0_BACKLOG.md`、`HARNESS_PAGE_API_MAPPING.md`、`docs/plans/current-roadmap.md` 的事实，项目已经具备分阶段交付条件：

- 第一阶段已完成 P0 主链路代码骨架
- 当前阶段以真实环境联调和专项收口为主
- 后续阶段可继续深化分镜专项、创作聚合页、剪辑增强和部署治理

因此，从业务交付路径来看，该项目具备逐步交付和逐段验收的现实基础。

## 4. 技术可行性

### 4.1 前端技术路线可行

当前前端采用“页面语义稳定 + `src/api/` 适配层翻译”的模式，其可行性已经被多轮联调证明：

- 字段命名冲突可在适配层集中处理
- 上传前置、媒体 URL、任务轮询等横切能力可在适配层封装
- 页面层不必因为后端命名调整而频繁改动

这套方式对应的核心决策已在 [ADR-001-frontend-first-adapter.md](file:///Users/xingyi/Desktop/打通前后端%202/docs/decisions/ADR-001-frontend-first-adapter.md) 中固化，因此技术路线明确且已被验证。

### 4.2 后端技术路线可行

当前后端基于 `FastAPI + router/service/model + PostgreSQL + task runtime` 的组织方式，已具备继续扩展的现实基础：

- 鉴权、项目、剧本、主体、分镜、创作、资产、剪辑等已有路由入口
- SSE、任务轮询、媒体上传/下载都已有首版落地
- 任务制已经覆盖主体提取、智能分镜、主体图生成、批量生成等关键链路

说明后端并非停留在“文档有、代码没有”的阶段，而是已有真实可复用能力。

### 4.3 文档治理路线可行

当前仓库已经从单一大文档阶段过渡到分层文档阶段：

- `HARNESS_DOC_INDEX.md` 负责总入口
- `docs/README.md` 负责文档中心入口
- `docs/architecture/*` 负责结构和约束
- `docs/plans/*` 负责路线图、模块进度和归档索引
- `CHANGELOG.md` 负责变更留痕

这意味着继续扩展“可行性 / 设计 / 进度 / 总结 / Todo”主文档，并不会破坏现有体系，而是顺着既有治理方向自然生长。

当前仓库已有以下工程资产可直接复用：

这表明本项目不存在“需要先补基础文档和基础结构才能开始推进”的阻塞。

### 5.2 开发方式具备可执行性

当前工程执行 SOP 已经足够明确：

- 先锁定页面闭环
- 再对齐前端输入输出和后端契约
- 再在适配层做映射
- 如有必要补后端能力或最小页面承接
- 最后做端到端验收和文档回写

这套方法在 `AGENTS.md` 中已有标准流程，因此项目具备稳定重复执行能力。

### 5.3 分阶段推进具备现实性

现有阶段分层清晰：

- 当前优先级 A：真实环境联调验收
- 当前优先级 B：分镜专项后续阶段
- 当前优先级 C：聚合页闭环验证
- 当前优先级 D：文档治理持续收口

这说明项目已经从“想到哪改到哪”走向“按阶段推进”，具备现实实施条件。

## 6. 成本与收益分析

### 6.1 继续沿现有路线推进的收益

- 保留现有前端产品表达，减少 UI 重做成本
- 通过适配层集中治理，降低未来字段变更成本
- 通过任务制统一承接长链路，降低异步能力接入成本
- 通过文档体系固化，降低后续接手和协作成本

### 6.2 若放弃当前路线的代价

如果放弃当前“前端优先 + 适配层治理 + 文档收口”路线，转而重做，会面临以下代价：

- 现有页面和专项接线成果大量报废
- 多轮联调积累的字段治理和经验沉淀失去价值
- 后续接手人需要重新梳理系统边界
- 短期内会大幅提高回归风险和工程不确定性

### 6.3 综合结论

继续沿当前路线推进的边际收益明显高于重建成本，因此在现阶段是更优方案。

## 7. 主要风险与约束

### 7.1 长任务稳定性风险

- 页面切换导致用户误以为任务中断
- 任务 `partial` 终态处理不一致
- 长任务实时进度与最终结果的承接还需继续验证

### 7.2 契约漂移风险

- 页面、适配层、后端若继续各写一套语义，后期维护成本会再次上升
- 新增模型/供应商能力若不入统一能力表，会再次出现多处硬编码

### 7.3 媒体链路风险

- 本地 / 私网地址无法被上游模型访问
- 视频参考、音频参考、公网 HTTPS 仍是敏感边界
- 上传、托管、原图、缩略图、下载链路必须持续治理

### 7.4 聚合页复杂度风险

- 资产库、创作页、剪辑页属于聚合型页面
- 如果过多聚合逻辑长期压在前端，后续维护成本会增加

### 7.5 真实环境验收风险

- 当前很多功能是“代码已打通，待真实联调”
- 真实账号、真实 API Key、真实媒体资源环境下仍可能暴露差异

---

## 8. 约束条件

当前继续推进必须遵守以下约束：

- 以前端页面为产品事实源，不随意改已有样式和主交互
- 以前端 `src/api/` 作为唯一集成适配层
- 以后端路由与 schema 作为接口事实源
- 以任务制优先承接长链路生成能力
- 以小文档优先、历史母本保留的方式治理文档

## 9. 阶段性结论

从业务、技术、工程和治理四个维度综合判断，当前项目具备明确可行性：

- 业务上有完整、连续且有价值的影视化 AI 工作流
- 技术上已有前端、后端、任务制和媒体链路的真实基础
- 工程上已形成可重复执行的 Harness SOP
- 治理上已具备统一入口、分层文档和历史归档机制

当前最正确的方向不是推翻重来，而是：

- 按当前文档体系继续收口设计与交接文档
- 按当前路线图继续推进真实联调与专项阶段
- 按当前事实源持续减少重复文档和重复逻辑
