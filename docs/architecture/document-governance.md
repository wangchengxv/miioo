# 文档治理与瘦身规则

## 1. 为什么要拆分

当前仓库已经形成较完整的 Harness 文档体系，但出现了两个问题：

- 同一个事实同时出现在 `PROJECT.md`、全局进度文档、mapping、backlog 中
- 大文件越来越长，AI 和人工都需要消耗更多上下文才能找到当前结论

因此本轮治理目标不是删历史，而是把“入口、当前、历史”拆开。

## 2. 新的三层模型

### 2.1 入口层

入口层只回答“先看什么”：

- `CLAUDE.md`
- `AGENTS.md`
- `HARNESS_DOC_INDEX.md`
- `docs/README.md`

### 2.2 当前层

当前层只维护当前阶段仍在推进的事实：

- `../需求分析文档.md`
- `可行性报告文档.md`
- `概要设计文档.md`
- `详细设计文档.md`
- `HARNESS_PAGE_API_MAPPING.md`
- `docs/plans/page-mapping-index.md`
- `HARNESS_P0_BACKLOG.md`
- `docs/plans/backlog-index.md`
- `docs/plans/current-roadmap.md`
- `docs/plans/module-progress.md`
- `docs/plans/项目进度文档.md`
- `docs/plans/开发总结文档.md`
- `docs/plans/功能模块Todo文档.md`
- `docs/plans/todo-index.md`
- `CHANGELOG.md`

### 2.3 历史层

历史层保留完整演进轨迹，但不再作为默认主入口：

- `HARNESS_P0_HISTORY.md`
- `miioo/frontend/PROJECT.md`
- `大模型官方文档/经验总结个人知识库/进度文档.md`

## 3. 写文档的默认顺序

1. 先判断是不是页面闭环事实
2. 再判断是不是当前优先级/阻塞
3. 再判断是不是系统设计或可行性结论
4. 再判断是不是模块进度、开发总结或模块 Todo
5. 最后才判断是否需要补历史母本

## 4. 大文件回写规则

只有以下情况，才回写历史大文件：

- 需要保留完整时间序列
- 需要沉淀跨阶段排障经验
- 该文档已经是外部协作方默认入口

否则新增信息优先写入当前层小文档。

## 5. 推荐维护粒度

- 原则类文件：保持稳定，少改
- 设计类文件：按阶段结论更新
- 路线图类文件：按阶段更新
- 进度类文件：按模块更新
- Todo 类文件：按执行状态更新
- 总结类文件：按阶段结论更新
- 变更日志：按日期追加
- 历史归档：按月或按专项索引，不再继续无限膨胀主入口

## 6. 当前新增主文档角色

为避免和既有文档重复，当前新增主文档的职责固定如下：

- `../需求分析文档.md`
  - 负责：项目背景、业务目标、系统范围、需求总稿
  - 不负责：模块级接口细节、逐日进度流水账
- `可行性报告文档.md`
  - 负责：业务可行性、技术可行性、工程可行性、风险和推进建议
  - 不负责：重复抄写详细设计或所有历史过程
- `概要设计文档.md`
  - 负责：系统层分层、模块职责、核心数据流、任务机制总览
  - 不负责：逐页面接口明细
- `详细设计文档.md`
  - 负责：模块级设计视角汇总、页面/适配层/后端能力关系
  - 不负责：取代 `HARNESS_PAGE_API_MAPPING.md` 和后端接口主文档
- `docs/plans/page-mapping-index.md`
  - 负责：页面映射的默认入口与主题导航
  - 不负责：重复保存完整历史母本
- `docs/plans/backlog-index.md`
  - 负责：阶段优先级与 backlog 的默认入口
  - 不负责：替代模块级 Todo
- `docs/plans/项目进度文档.md`
  - 负责：接手视角的当前项目状态与阅读建议
  - 不负责：逐日 changelog
- `docs/plans/开发总结文档.md`
  - 负责：经验沉淀、排障结论、长期有效的工程判断
  - 不负责：阶段优先级清单
- `docs/plans/功能模块Todo文档.md`
  - 负责：按模块组织、细到页面/接口层的执行清单
  - 不负责：替代 `HARNESS_P0_BACKLOG.md` 的阶段优先级角色
- `docs/plans/todo-index.md`
  - 负责：Todo 子文档的默认入口
  - 不负责：替代根级 Todo 固定入口
