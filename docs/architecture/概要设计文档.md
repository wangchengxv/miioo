它不是接口明细，也不是逐日进度记录，而是一份面向接手开发者、项目 owner 和联调同学的系统级设计说明书。

## 2. 设计目标

当前系统设计目标不是重写全部前后端，而是在已有产品事实基础上完成真实业务闭环与工程治理收口。

概要设计层面的总目标包括：

- 保留现有前端产品体验和工作流顺序
- 用 `frontend/src/api/` 统一承接双端字段与结构翻译
- 用后端 router/service/model 承接稳定业务能力
- 用统一任务体系承接长链路生成型能力
- 用 Harness 文档系统承接事实、阶段、经验和交接

## 3. 系统范围

当前系统面向影视创作者，覆盖的主要业务模块如下：

- 登录与当前用户
- 首页与通知
- 项目列表与新建项目
- 全局设定
- 剧本与分集
- 主体
- 分镜
- 创作页
- 资产库
- 剪辑成片
- API 配置
- 通用任务体系
- 媒体上传、下载与资源托管

系统的核心业务主链路如下：

```text
首页/登录
  -> 项目列表 / 新建项目
  -> 项目工作流
     -> 全局设定
     -> 剧本与分集
     -> 主体
     -> 分镜
     -> 创作页
     -> 资产库
     -> 剪辑成片
```

## 5. 单一事实源设计

为了避免“页面一套、后端一套、文档又一套”的情况，系统采用三类单一事实源：

### 5.1 产品事实源

- 前端页面、组件和交互流程
- 主要目录：`miioo/frontend/src/pages/`、`miioo/frontend/src/components/`

### 5.2 接口事实源

- 后端路由、schema、响应结构、鉴权规则、任务状态规则
- 主要入口：`miioo/backend/BACKEND_API_DOC.md`、`miioo/backend/app/routers/`

### 5.3 集成事实源

- 前端 `src/api/` 适配层
- 负责前后端字段翻译、请求编排、响应归一化、任务轮询封装、上传前置

这三类事实源分别解决：

- 页面应该长什么样
- 后端实际上提供什么能力
- 页面和后端如何稳定对接

## 6. 运行分层设计

### 6.1 页面层

目录：

- `miioo/frontend/src/pages/`
- `miioo/frontend/src/components/`

职责：

- 承接产品交互与视觉结构
- 承接最小必要状态展示
- 保持页面语义稳定

禁止事项：

- 页面中直接写真实请求
- 页面中散落后端命名兼容逻辑
- 页面直接处理 token、上传细节、轮询细节

### 6.2 前端适配层

目录：

- `miioo/frontend/src/api/`

职责：

- 请求字段映射
- 响应视图模型归一化
- 上传前置与媒体 URL 标准化
- 模型能力翻译
- 任务状态查询与封装

该层是当前前后端联调的核心集成层。

### 6.3 后端接口层

目录：

- `miioo/backend/app/routers/`

职责：

- 暴露稳定的路由、状态、错误语义
- 提供必要兼容字段或聚合接口
- 管理任务接口、上传下载接口、生成入口

### 6.4 后端服务层与模型层

目录：

- `miioo/backend/app/services/`
- `miioo/backend/app/models/`
- `miioo/backend/app/schemas/`

职责：

- 执行核心业务逻辑
- 封装上游模型调用
- 实现数据库持久化
- 推进长任务执行流

### 6.5 文档治理层

目录：

- 根级 Harness 文档
- `docs/`

职责：

- 定义原则与边界
- 维护阶段目标与当前状态
- 留存经验和历史
- 为后来者提供统一接手入口

## 8. 核心数据流设计

系统核心数据流如下：

```text
页面交互
  -> frontend/src/api
  -> backend router
  -> backend service
  -> model/db/task runtime
  -> backend router response
  -> frontend/src/api normalize
  -> 页面状态回显
```

这条数据流的关键设计点在于：

- 页面不直接依赖后端字段
- 适配层统一做翻译和归一化
- 后端统一输出稳定契约
- 页面最终只消费自己需要的视图模型

## 9. 统一任务机制设计

当前长链路生成能力统一优先使用任务制。

已纳入任务制的重要链路包括：

- 主体提取
- 智能分镜
- 单主体图片生成
- 批量主体生成
- 创作页长时任务
- 剪辑导出任务

统一原则如下：

- 后端优先返回 `taskId`
- 前端通过任务状态接口轮询
- 页面切换后任务继续执行
- 页面重新进入后可恢复查看
- `completed / failed / partial / cancelled` 都是受支持终态

任务体系的存在，是当前系统能够承接长时 AI 能力的关键基础设施。

## 10. 媒体与资源链路设计

### 10.1 资源原则

- 本地文件不能直接作为长期事实源
- blob URL 只用于前端临时预览
- 真实资源必须通过后端受控 URL 承接

### 10.2 上传设计

- 页面上传本地文件
- 适配层先调用上传接口
- 后端保存为受管资源
- 生成类接口消费真实托管 URL

### 10.3 下载设计

- 优先走后端受控下载接口
- 原图、缩略图、托管图语义应区分清楚
- 对第三方返回的临时地址需要必要时做本地托管

### 10.4 公网访问设计

- 上游模型若需要访问媒体资源，后端需保证转换成可访问公网 URL
- 当前视频参考、音频参考与 HTTPS 是重点边界

## 12. 关键设计约束

当前概要设计必须长期坚持以下约束：

- 前端页面是产品事实源，不轻易改样式和主交互
- 适配层是唯一集成层，不让后端细节上浮到页面
- 后端保持业务语义清晰，不直接耦合前端局部 UI 表达
- 文档不重复抄写同一事实，避免出现多套真相
- 历史文档保留，但默认入口以小文档为主

## 13. 当前阶段的系统性结论

当前系统不是“功能还没开始”的状态，而是：

- 主要业务模块已经具备第一版真实代码主干
- 系统级设计方向已经稳定
- 当前主要工作重心转入真实环境验收、专项深化和文档治理

因此，概要设计层面最重要的不是再发明新结构，而是：

- 继续沿当前结构补齐缺口
- 继续减少跨层职责混乱
- 继续把当前治理方式固化成接手友好的主文档体系
