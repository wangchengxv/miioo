# 架构文档说明

## 1. 目录定位

本目录用于承接“需求之后、代码之前”的设计层文档，默认回答四个问题：

- 我们到底要做什么
- 为什么当前路线可行
- 系统应该怎么分层
- 模块应该如何落地

## 2. 默认阅读顺序

1. `../需求分析文档.md`
2. `可行性报告文档.md`
3. `概要设计文档.md`
4. `详细设计文档.md`
5. `document-governance.md`

## 3. 四类文档分工

| 文档 | 负责回答的问题 | 不负责什么 |
|---|---|---|
| `../需求分析文档.md` | 产品背景、业务目标、系统范围、阶段目标 | 逐接口明细、逐日流水账 |
| `可行性报告文档.md` | 为什么当前路线值得继续、风险和推进方式是否可行 | 重复抄写详细设计或所有历史过程 |
| `概要设计文档.md` | 系统分层、模块职责、核心数据流、任务机制总览 | 逐页面接口和字段细节 |
| `详细设计文档.md` | 模块级落地方式、页面/适配层/后端能力关系、关键风险 | 取代页面映射事实或后端接口主文档 |

## 4. 与其它文档的关系

- 页面闭环事实继续看 `../HARNESS_PAGE_API_MAPPING.md` 和 `../docs/plans/page-mapping-index.md`
- 阶段优先级继续看 `../HARNESS_P0_BACKLOG.md` 和 `../docs/plans/backlog-index.md`
