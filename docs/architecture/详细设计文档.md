# 详细设计文档

## 1. 文档定位

本文档用于承接 `miioo` 当前各核心模块在“页面行为、适配层映射、后端契约、任务机制、状态承接、风险边界”上的详细设计。

它是对 [概要设计文档.md](file:///Users/xingyi/Desktop/打通前后端%202/docs/architecture/%E6%A6%82%E8%A6%81%E8%AE%BE%E8%AE%A1%E6%96%87%E6%A1%A3.md) 的进一步展开，但不替代：

- `HARNESS_PAGE_API_MAPPING.md` 的页面闭环事实
- `miioo/backend/BACKEND_API_DOC.md` 的后端接口主事实
- `.trae/documents/*.md` 的专项实施计划

本文档的目标是给后来者一份“设计视角的结构化总汇”，而不是要求对方去几十份文档里自行拼上下文。

本文档不负责：

- 替代 `HARNESS_PAGE_API_MAPPING.md` 保存逐页面闭环事实
- 替代 `miioo/backend/BACKEND_API_DOC.md` 保存接口主契约
- 承担逐阶段优先级和执行清单角色

## 2. 详细设计总原则

当前详细设计统一遵守以下约束：

- 页面继续保持前端产品语义
- 前端 `src/api/` 是唯一集成适配层
- 后端 router/service/model 承接真实业务能力
- 生成型长任务优先纳入任务制
- 文档回写是交付的一部分，不是附加项

通用映射规则包括：

- `desc -> description`
- `ratio -> aspect_ratio`
- `style -> visual_style`
- `shot -> storyboard`
- `char -> character`
- 集数展示文案 -> `episode_id`

## 3. 登录与鉴权详细设计

### 3.1 设计目标

- 支持验证码登录、密码登录、退出登录、当前用户恢复
- 在 `401` 场景下自动尝试刷新 token 并重试原请求
- 对匿名浏览和需登录动作做清晰门禁

### 3.2 页面与组件

- `Home.jsx`
- `LoginModal.jsx`
- `AccountMenu.jsx`
- `ProfileModal.jsx`

### 3.3 前端适配层

- `src/api/auth.js`
- `src/api/user.js`

### 3.4 后端能力

- `/api/auth/send-code`
- `/api/auth/verify-code-login`
- `/api/auth/login`
- `/api/auth/refresh`
- `/api/auth/logout`
- `/api/auth/me`
- 微信扫码与确认相关接口

### 3.5 详细设计要点

- 登录成功后统一归一会话结构
- 当前用户对象归一成页面统一可消费字段
- `Home.jsx` 在刷新后先恢复会话，再恢复页面上下文
- 未登录态允许浏览部分页面，但在关键动作处拉起登录弹窗

### 3.6 当前风险

- 真实短信环境与本地开发验证码环境都需要继续联调
- 密码设置 / 重置闭环仍不完整
- 微信绑定与绑定手机仍有后续工作

## 4. 项目列表与全局设定详细设计

- 用户能查看项目、创建项目、进入项目、修改项目基础信息

## 5. API 配置详细设计

### 5.1 设计目标

- 为图片、视频、文本、配音能力提供统一 provider 和模型配置入口
- 保持页面仍以“一个配置弹窗”的产品形态承接，内部由适配层做聚合编排

### 5.2 页面与组件

- `ApiConfigModal.jsx`
- `Home.jsx`

### 5.3 前端适配层

- `src/api/config.js`

### 5.4 后端能力

- `providers` CRUD
- built-in provider setup 路由
- `models` CRUD
- provider 测试连接接口

### 5.5 详细设计要点

- 页面外观保持单模块配置体验
- 内部通过聚合适配层把多 provider、多模型、多默认值编排成单一视图模型
- 模型真实 `provider_type / provider_name` 作为下游入口判断事实源
- 登录后 `hasApiConfigured` 会影响项目和创作关键入口门禁

### 5.6 当前风险

- 真实 provider key 环境下仍需逐个验证
- 部分 provider 的能力边界与默认模型仍需继续收口

## 6. 剧本与分集详细设计

### 6.1 设计目标

- 支持正式分集的读取、创建、保存、上传、删除
- 支持整稿工作区 `chat -> split-preview -> finalize`
- 支持流式生成与主体提取入口

### 6.2 页面与组件

- `ScriptPage.jsx`

### 6.3 前端适配层

- `src/api/episode.js`
- `src/api/projectScript.js`

### 6.4 后端能力

- `episodes` 相关 CRUD、generate、stream、upload
- `script-workspace` 相关接口
- `subjects/extract`

### 6.5 详细设计要点

- 当前页面保留原有工作区语义，但底层已切换为真实 episode 对象和 `episode_id`
- 当没有正式分集时，整稿链路走 `script-workspace`
- 目标集数统一映射为 `episode_count`
- 默认文本模型从 API 配置页默认模型动态刷新
- “开始提取主体”前会先保存当前剧本，再转入任务制主体提取

### 6.6 当前风险

- 真实 SSE 场景和整稿拆分场景仍需重点联调
- 目标集数对模型输出结构的影响仍需持续观察

- 支持图片、视频、配音三条创作链路
- 支持统一素材入口、上传前置、结果管理、详情查看、批量操作

## 12. 通用任务与媒体链路详细设计

### 12.1 通用任务

统一要求：

- 提交长任务后立即返回 `taskId`
- 前端轮询统一任务状态接口
- 支持中间进度或部分结果回显
- 支持 `partial` 终态
- 支持切页后继续执行

### 12.2 媒体链路

统一要求：

- 本地文件先上传后参与业务处理
- blob URL 只做前端预览
- 上游模型消费真实可访问 URL
- 原图、缩略图、下载图语义区分清楚

### 12.3 当前风险

- `PUBLIC_BASE_URL` 与 HTTPS 要求仍是视频链路重点
- 本地 / 私网媒体地址若未转公网可访问地址，会直接导致上游失败

- 新事实优先写入当前层小文档
- 只有需要保留完整历史链路时，才回写历史母本

## 14. 当前详细设计结论

从详细设计层面看，当前系统已经具备以下特征：

- 各主模块都有明确页面、适配层、后端能力与数据承接方式
- 关键跨模块能力已收口到任务体系、模型能力事实源和媒体链路治理
- 当前剩余工作重点集中在真实环境联调、专项深化和交接文档完善

因此，后续实现不需要再重新发明整体设计，而应继续围绕现有结构补齐剩余缺口与验收闭环。
