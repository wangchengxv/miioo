# 媒体链路交接模板

## 1. 目标

本模板用于在 `MEDIA-BPLUS-*` 阶段性完成后，快速整理可交接包。

它只回答：

- 现在做到哪里
- 下一步该做什么
- 哪些文档和证据必须一起交给接手者

## 2. 适用范围

适用于以下场景：

- 一个 `MEDIA-BPLUS-*` 任务包完成阶段性收口
- 准备从设计阶段移交到实现阶段
- 准备从实现阶段移交到联调、灰度或验收阶段
- 需要把当前结论交给下一个协作者继续推进

推荐与以下文档配合使用：

- [MEDIA-BPLUS-70-文档回写闭环与验收矩阵实现设计.md](file:///Users/xingyi/Desktop/迭代一版/docs/plans/MEDIA-BPLUS-70-文档回写闭环与验收矩阵实现设计.md)
- [media-acceptance-checklist.md](file:///Users/xingyi/Desktop/迭代一版/docs/runbooks/media-acceptance-checklist.md)
- [项目进度文档.md](file:///Users/xingyi/Desktop/迭代一版/docs/plans/项目进度文档.md)

## 3. 最小交接包

任意阶段交接时，至少应包含：

1. 当前专项设计稿
2. 对应 backlog 入口
3. 对应实施清单入口
4. `module-progress.md` 摘要
5. `项目进度文档.md` 时间线
6. `CHANGELOG.md` 留痕

## 4. 完整交接包

若已进入真实实现、灰度或验收阶段，建议额外包含：

1. 相关 runbook
2. 验收结果
3. 验证证据
4. 风险与回滚路径
5. 下一步建议

## 6. 推荐写法

- `一句话状态` 建议写成：
  - `设计已收口，待进入实现`
  - `代码已完成，待用户手工联调`
  - `灰度已完成，待最终验收`
- `当前事实源` 优先写当前层小文档，不要只给历史母文档。
- `验收状态` 若依赖你手工验证真实环境，明确写：
  - `代码与文档已就绪，真实环境联调待用户手工验证`

## 7. 不建议的交接方式

- 只在聊天里口头说明，不留文档
- 只给一个大文件，不给当前专项设计稿
- 只说“都做完了”，不写未完成项和风险
- 不写下一步建议，导致接手者需要重新判断上下文
