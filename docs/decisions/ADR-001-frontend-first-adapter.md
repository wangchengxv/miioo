# ADR-001：前端优先，适配层承接双端契约

## 状态

已批准（2026-06-10）

## 背景

当前项目的前端页面、交互结构和 UI 已经先行成型，而后端接口是在后续联调过程中逐步补齐的。若直接要求页面配合后端字段重命名或重做交互，会破坏产品事实源，并增加回归风险。

## 可选方案

### 方案 A：页面直接适配后端字段

优点：

- 看起来改动集中在页面里

缺点：

- 后端字段散落到组件树
- 同一概念容易在多个页面重复翻译
- 后续字段变更时维护成本高

### 方案 B：后端完全改成前端口径

优点：

- 页面侧最省改动

缺点：

- 容易把单页历史命名固化成全局后端契约
- 多模块时会污染后端模型边界

### 方案 C：保留页面口径，统一由 `src/api/` 适配层翻译

优点：

- 页面语义稳定
- 映射集中，方便治理
- 后端仍可保持相对一致的业务模型

缺点：

- 适配层会变成联调核心，需要持续维护

## 最终决策

采用方案 C。前端页面继续作为产品事实源，前后端契约翻译统一收口到 `miioo/frontend/src/api/`。

## 落地后果

- 页面组件禁止直接书写真实请求和后端字段兼容逻辑
- 新增映射必须优先回写 `HARNESS_PAGE_API_MAPPING.md`
- 若适配层开始出现大面积扭曲，再评估是否升级为双端统一改造
